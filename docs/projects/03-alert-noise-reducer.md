# 项目 3：告警降噪器

## 背景

生产事故中，一个根因常常引发几十条甚至上百条告警。告警降噪的目标不是隐藏风险，而是把重复告警合并成可处理事件。

## 输入

- 告警名称。
- 服务名。
- 主机名或 Pod 名。
- 严重级别。
- 触发时间。
- 标签。
- 可选：服务依赖关系、最近变更记录。

## 规则

第一版使用可解释规则：

- 同一服务、同一告警名、5 分钟内合并。
- 同一依赖链路上的多个服务告警，按拓扑聚合。
- 资源类告警和应用错误告警同时出现时，保留二者但归为同一事件。
- 最近变更窗口内的告警标注“可能与变更相关”，不直接判定根因。

## 输出

```text
事件 E-20260701-001
影响服务：order-api, payment-api
告警数量：23
首个告警：10:13
最高级别：critical
可能线索：payment-api 10:08 有发布变更；order-api 错误率随 payment-api 超时上升
建议动作：检查 payment-api 发布版本和下游连接池
```

## 验收

- 输入 50 条模拟告警，输出少于 10 个事件。
- 每个事件保留原始告警列表，不能丢证据。
- 有降噪率统计。
- 有误合并案例复盘。

## 简历表达草稿

实现基于时间窗口、服务标签和依赖关系的告警聚合原型，将重复告警压缩为事件视图，并保留原始证据链、降噪率和误合并复盘，降低值班人员排障入口复杂度。
