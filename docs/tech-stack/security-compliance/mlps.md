## 官方知识地图

等级保护资料不是一份“三级检查表”，而是一套分层体系：

```text
法律与行政法规
  -> 明确网络运营者的安全保护义务
  -> 管理办法和主管部门要求
  -> 定级指南：保护哪个对象、定几级
  -> 基本要求：应建设哪些技术和管理控制
  -> 安全设计与实施指南：控制怎么规划和落地
  -> 测评要求与过程指南：怎么检查控制是否有效
  -> 监督检查与持续运营：变化后如何继续满足要求
```

本文覆盖：

1. 等保是什么，以及它与数据安全、个人信息保护、关键信息基础设施保护的边界。
2. 五个等级怎么理解，为什么不能先拍脑袋说“我要做三级”。
3. 定级、备案、建设整改、等级测评和监督检查五个阶段。
4. 技术控制、管理控制和云计算等扩展场景。
5. 资产、控制、证据、问题和整改如何形成持续闭环。
6. AIOps 如何帮助做资产发现、基线漂移、日志覆盖、漏洞治理和证据保鲜。

## 场景开场

公司准备把新的 AIOps 平台投入生产。平台能读取监控、日志、告警、CMDB 和历史故障记录，还能调用自动化 Runbook。

客户问：“这个平台过等保三级了吗？”

云厂商说：“我们的云平台已经有等保测评报告。”开发说：“应用部署在这个云上，应该可以直接复用吧？”运维说：“防火墙、堡垒机、日志平台和杀毒软件都有，是不是买齐产品就行？”

真正要先回答的是：

- 被保护对象到底是哪套业务，边界到哪里？
- 它被破坏后会损害谁，危害程度有多大？
- 云厂商、平台团队、应用团队和安全团队分别负责什么？
- 账号、网络、主机、应用、数据、日志、备份和应急控制是否真的在运行？
- 测评当天合格后，下个月发布新功能、增加公网入口或更换云服务，证据还成立吗？

等级保护解决的不是“拿一张证书”，而是让重要程度不同的网络和系统承担相匹配、可验证、可持续的安全保护责任。

## 一句话人话版

等保是先按系统受破坏后的影响定级，再按等级建设、测评和持续运行安全控制的一套国家网络安全保护制度。

## 小白可能会问

- “等保三级”是公司等级、产品等级，还是某个具体系统的等级？
- 系统部署在已经通过测评的云平台上，业务应用还要不要做？
- 安装防火墙、WAF、堡垒机、日志审计和杀毒软件，是不是就能通过？
- 测评报告出来后，为什么还要持续整改、监控和演练？

## 为什么 AIOps 工程师要学等保

AIOps 平台往往拥有很宽的读取权限，能集中看到主机、容器、数据库、日志、告警和用户行为，有些平台还可以执行生产动作。它既是安全证据的重要来源，也可能成为高权限攻击入口。

运维和 AIOps 岗位经常需要参与：

- 梳理服务器、网络设备、云资源、Kubernetes 集群、数据库和中间件资产。
- 证明账号最小权限、管理员操作审计、补丁和恶意代码防护有效。
- 证明安全日志完整、时间一致、可检索，并满足适用的留存要求。
- 跟踪漏洞、弱口令、基线偏差和配置变更是否按时闭环。
- 证明备份不只是“任务成功”，而是真的能恢复。
- 为测评、监督检查和事件响应快速提供可信证据。

如果不理解等保，容易把工作做成“临测评前补截图”。理解以后，才能把控制要求变成日常指标、告警、工单、Runbook 和审计记录。

## 等保是什么

《网络安全法》第二十三条规定国家实行网络安全等级保护制度，并要求网络运营者落实内部制度与责任人、攻击和入侵防范、网络运行和安全事件监测记录、数据分类与备份加密、应急处置等义务。

“等级”针对的是具体等级保护对象，不是给公司或某个安全产品贴一个永久标签。对象可以是一套承载明确业务的网络或信息系统，也可能是云计算、移动互联、物联网、工业控制等场景下划分出的保护对象。

### 关键术语

| 术语 | 人话解释 | 为什么重要 |
|---|---|---|
| 等级保护对象 | 本次定级和保护的具体网络、系统或平台边界 | 边界错了，资产、控制和测评范围都会错 |
| 网络运营者 | 建设、运营网络或通过网络提供服务并承担安全义务的主体 | 外包和上云通常不能转移主体责任 |
| 安全保护等级 | 对象被破坏后的危害对象和危害程度对应的等级 | 决定建设和测评要求的强度 |
| 定级 | 分析对象边界、受侵害对象和危害程度，提出等级 | 不能用“预算够不够”决定等级 |
| 备案 | 按适用要求向公安机关提交定级等材料 | 不是把测评报告上传到任意平台 |
| 等级测评 | 由符合规定条件的机构按标准检查安全控制 | 测评发现问题不等于测评机构替你整改 |
| 测评证据 | 能证明控制在特定对象、时间和范围内有效的记录 | 没有证据，口头说“已经做了”通常无法验证 |
| 高风险问题 | 可能对系统造成重大安全影响、需要优先处理的问题 | 不能只盯总分而忽略关键风险 |

“MLPS”常被用作等级保护英文简称，官方国家标准英文使用的是 classified protection of cybersecurity。面试或文档里最好先写中文全称，再说明业内简称，避免只丢一个缩写。

## 等保不是什么

### 不是采购清单

标准描述的是控制目标。防火墙、WAF、EDR、堡垒机、数据库审计、日志平台等产品只是可能的实现方式。已有平台能满足控制目标并提供有效证据时，不应为了“凑品牌”重复采购；反过来，设备买了但策略为空、日志没接、告警没人处理，也不等于控制有效。

### 不是一次性认证

账号、漏洞、资产、配置、供应链、数据流和业务边界一直在变化。测评报告只反映特定范围和时间点的状态，不能证明未来一直安全。

### 不等于全部合规

等保重点是网络和系统的分级安全保护。涉及个人信息、重要数据、数据出境、关键信息基础设施、密码应用、行业监管时，还要同时识别《个人信息保护法》《数据安全法》《关键信息基础设施安全保护条例》及相关行业规则。通过等保测评不能自动推导出这些义务都已完成。

### 云平台通过不等于云上业务自动通过

云服务商通常负责其云基础设施和约定服务范围，租户仍要负责业务定级、账号、网络策略、操作系统或容器、应用、数据、密钥、日志、备份和应急等责任。实际边界要以云服务模式、合同、责任矩阵和测评范围为准。

## 五个安全保护等级

定级主要看两个维度：对象被破坏后会侵害谁，以及危害有多严重。下面是帮助入门的概括，正式定级必须以 `GB/T 22240-2020` 和主管部门要求为准。

| 等级 | 受破坏后的影响概念 | 常见理解 |
|---|---|---|
| 第一级 | 主要损害公民、法人和其他组织的合法权益，影响有限 | 自主保护，安全要求相对基础 |
| 第二级 | 对合法权益造成严重损害，或损害社会秩序、公共利益 | 大量普通联网业务可能从这里开始评估 |
| 第三级 | 对合法权益造成特别严重损害，或严重损害社会秩序、公共利益，或损害国家安全 | 重要业务系统常重点评估，技术和管理要求明显提高 |
| 第四级 | 对社会秩序、公共利益造成特别严重损害，或严重损害国家安全 | 适用于极重要对象，要求更严格 |
| 第五级 | 对国家安全造成特别严重损害 | 由国家指定专门部门监督，普通企业很少自行涉及 |

不能从“用户数量”“是不是政府项目”“客户要求三级”单独推出等级。正式定级要先划清对象边界，再识别业务功能、服务范围、数据、依赖关系、受侵害对象和危害程度，并按要求完成专家评审、主管部门审核或备案沟通。

## 定级最容易错在哪里

### 把整个公司定成一个对象

一个公司可能同时有官网、办公系统、交易系统、生产控制系统和 AIOps 平台。它们的业务边界、用户、数据和危害不同，通常不能为了省事全部揉成一个对象。

### 为了省成本故意低定级

等级来自客观危害，不是预算选项。低定级会导致控制强度与真实风险不匹配，也可能在备案、监督检查或事故后暴露问题。

### 为了投标直接高定级

等级过高会增加建设、测评和持续运营成本。投标要求、客户合同和法定定级要分开管理，不能用一句“做三级更保险”替代边界与危害分析。

### 忽略外部依赖

公网入口、第三方登录、短信、支付、云数据库、对象存储、CI/CD、监控平台、远程运维和供应商接入都会影响边界与风险。拓扑图只画三台服务器，通常无法支持真实定级和测评。

## 等保工作的五个阶段

常见主流程是定级、备案、安全建设整改、等级测评和监督检查。工程上还应把它扩展成持续循环：

```text
资产和业务梳理
  -> 确定保护对象与边界
  -> 初步定级、专家评审和主管部门审核
  -> 按要求备案
  -> 差距分析与安全建设整改
  -> 等级测评
  -> 问题整改与复核
  -> 日常监控、变更管理和应急演练
  -> 边界或风险变化时重新评估
```

### 1. 定级

是什么：确定保护对象和安全保护等级。

为什么需要：不同重要程度的对象不能使用同一套最低防护。

怎么工作：分析对象边界、业务信息安全和系统服务安全受到破坏后的影响，再提出等级并按要求评审审核。

怎么验证：应有对象描述、边界、拓扑、资产和依赖、数据类型、危害分析、定级报告及评审意见。

坏了怎么查：如果业务名称、云账号、域名、集群、数据库和数据流对不上，通常说明边界材料已经过期。

### 2. 备案

是什么：第二级以上对象通常按管理要求向公安机关办理备案，具体材料和流程以属地及行业要求为准。

为什么需要：让监管部门掌握对象、等级和责任主体，并检查定级是否合理。

怎么工作：准备定级、拓扑、组织责任和其他要求材料，提交受理机关审核；对象或等级发生重要变化时评估是否重新备案。

怎么验证：保留备案证明、提交材料版本、受理记录、整改通知和变更记录。

坏了怎么查：最常见的是系统名称不一致、备案边界与生产不一致、联系人离职、云资源迁移后材料未更新。

### 3. 安全建设整改

是什么：按相应等级的基本要求和扩展要求建设技术与管理控制，并修复差距。

为什么需要：测评不是替代建设；没有长期运行的控制，临时截图没有意义。

怎么工作：建立差距台账，明确责任人、方案、风险、计划、验证方法和残余风险，再按变更流程实施。

怎么验证：配置、日志、工单、制度、审批、演练、恢复记录和责任人访谈应能相互印证。

坏了怎么查：如果整改只写“已完成”却没有验证记录，或者制度和系统配置相互矛盾，控制还没有真正闭环。

### 4. 等级测评

是什么：按测评要求对技术和管理控制进行访谈、检查、测试和综合判断。

为什么需要：由独立视角验证控制是否覆盖对象、配置正确并持续有效。

怎么工作：测评准备、方案确认、现场测评、问题确认、报告和整改复核。第二级以上对象按适用要求开展测评，第三级以上对象的周期要求通常更严格；实际周期以现行法规、行业和属地要求为准。

怎么验证：测评范围、系统版本、证据时间、测试授权、问题清单和报告结论应一致。

坏了怎么查：只追求某个分数、隐瞒高风险问题、把不在范围内的云平台报告当成本系统证据，都会让结论失真。

### 5. 监督检查与持续运营

是什么：主管部门监督检查，加上运营单位自己的持续安全管理。

为什么需要：安全状态会随发布、扩容、漏洞、人员和供应商变化。

怎么工作：持续维护资产、账号、漏洞、日志、备份、应急、供应链和整改台账，定期自查并配合监督检查。

怎么验证：查看最近一段时间的趋势和闭环记录，而不是只有测评周的证据。

坏了怎么查：如果所有制度都同一天签发、所有截图都来自测评前一天，而日常工单和日志为空，说明控制可能没有持续运行。

## 技术要求：五个安全域

`GB/T 22239-2019` 的通用安全要求包含技术和管理两部分。技术侧可以先记住五个安全域：

| 安全域 | 解决什么 | 常见控制 | 有效证据 | 常见失效 |
|---|---|---|---|---|
| 安全物理环境 | 保护机房、设备和介质 | 门禁、防火、防水、防雷、电力、温湿度、介质管理 | 门禁记录、巡检、环境告警、维保和演练 | 只有制度，没有巡检和告警处置 |
| 安全通信网络 | 保护网络架构和传输 | 冗余、可信通信、链路和设备防护 | 拓扑、路由、链路监控、加密配置 | 单链路、明文管理、拓扑与生产不一致 |
| 安全区域边界 | 控制不同区域之间的访问 | 分区分域、访问控制、入侵防范、恶意代码防范、安全审计 | 防火墙策略、流量日志、IDS/WAF 告警、变更单 | 任意互通、长期临时放行、告警无人处理 |
| 安全计算环境 | 保护主机、应用、数据库、容器和数据 | 身份鉴别、访问控制、审计、入侵防范、资源控制、数据保护 | 账号、基线、补丁、EDR、审计、备份恢复记录 | 共享账号、弱口令、过权、失补丁、日志缺失 |
| 安全管理中心 | 集中管理、审计和联动 | 系统管理、审计管理、安全管理、集中管控 | 堡垒机、SIEM、集中日志、告警工单、权限审批 | 平台在线但数据没接全、规则没维护、无人值守 |

“一个中心、三重防护”常用来概括安全管理中心，以及安全通信网络、安全区域边界、安全计算环境的体系化防护。安全物理环境和管理要求仍然不能忽略。

## 管理要求：五个管理域

| 管理域 | 要回答的问题 | 典型证据 |
|---|---|---|
| 安全管理制度 | 有没有成体系、可执行、定期评审的制度和流程 | 制度清单、版本、审批、发布与评审记录 |
| 安全管理机构 | 谁决策、谁负责、谁执行、谁监督 | 组织架构、职责、会议纪要、授权和汇报机制 |
| 安全管理人员 | 入职、在岗、调岗、离职和外部人员如何管 | 背景审查、保密协议、培训、权限回收记录 |
| 安全建设管理 | 新系统、开发、采购、外包、上线和验收怎么控 | 安全需求、设计评审、代码与漏洞检测、验收材料 |
| 安全运维管理 | 资产、配置、账号、漏洞、日志、备份、变更和应急怎么持续运行 | CMDB、工单、巡检、审计、备份恢复、演练和复盘 |

管理制度不能只写“应当定期检查”。要明确责任角色、触发条件、频率、输入、操作步骤、输出证据、异常升级和记录保存位置。

## 扩展场景

等保 2.0 的基本要求在通用要求之外，还针对不同技术场景给出扩展要求。是否适用取决于保护对象真实使用的技术。

### 云计算

重点关注云服务商与云租户责任边界、虚拟化隔离、镜像和快照、云管理面、云资源回收、云审计日志、数据迁移和退出机制。

新手最容易错在：把云平台的测评报告当作租户应用的全部证据。正确做法是建立共享责任矩阵，逐项写清“云厂商提供什么、租户配置什么、谁提供哪份证据”。

### 移动互联

重点关注移动终端、移动应用、无线通信、客户端完整性、恶意代码、敏感权限、接口认证和终端丢失风险。

### 物联网

重点关注感知节点身份、设备生命周期、固件升级、默认口令、通信保护、网关边界和大量弱终端被控制后的影响。

### 工业控制系统

重点关注生产连续性、控制网络分区、工业协议、工程师站、远程维护、白名单和安全变更。工业控制不能照搬互联网系统“随时扫描、随时重启”的方式，测试必须考虑生产安全和停机窗口。

## 关键控制怎么落地

### 资产与边界

是什么：知道保护对象里有什么、在哪里、归谁管、暴露给谁。

为什么需要：未知资产无法打补丁、收日志或控制访问。

怎么工作：统一业务、域名、IP、云资源、主机、容器、数据库、中间件、账号、证书和第三方依赖标识，持续和真实环境对账。

怎么用：CMDB、云 API、Kubernetes API、网络扫描和人工确认互相校验。

坏了怎么查：公网域名找不到责任人、云账单有资源但 CMDB 没有、Kubernetes namespace 无业务归属，都是边界失真信号。

### 身份鉴别与访问控制

是什么：确认“谁在访问”，并只授予完成工作所需权限。

为什么需要：共享管理员账号、长期凭证和过度授权会让攻击和追责都变容易。

怎么工作：唯一账号、强认证、MFA、RBAC、最小权限、特权账号审批、定期复核和离职回收形成完整生命周期。

怎么用：目录服务、IAM、堡垒机、数据库账号、云角色和 Kubernetes RBAC 都要纳入同一责任视图。

坏了怎么查：账号长期无人登录、责任人已离职、管理员不能追到个人、服务账号可以交互登录、权限复核只有签字没有变更记录。

### 网络分区与边界访问

是什么：按业务和信任级别划分区域，只允许必要方向、端口和来源访问。

为什么需要：一台边缘主机失陷后，不应直接横向访问数据库和管理面。

怎么工作：网络区域、安全组、防火墙、NetworkPolicy、入口网关和远程运维通道共同实施访问路径。

怎么用：每条放行应关联业务、责任人、期限和变更单，临时规则到期自动提醒。

坏了怎么查：`0.0.0.0/0` 管理端口、any-any 规则、测试网直连生产、已下线业务规则仍保留。

### 漏洞、补丁和安全配置

是什么：持续发现软件缺陷、弱配置和不必要暴露，并按风险闭环。

为什么需要：一次扫描只能发现当时已知问题，不能替代修复和验证。

怎么工作：资产发现、漏洞扫描、情报、风险排序、工单、变更、修复、复测和例外审批形成闭环。

怎么用：把漏洞与资产重要性、是否公网、是否有利用、业务影响和补偿控制一起排序。

坏了怎么查：扫描资产数低于 CMDB、严重漏洞长期无责任人、关闭工单但版本没变、例外没有到期时间。

### 日志审计与时间同步

是什么：记录关键登录、授权、配置变更、管理操作、安全事件和业务异常，并能集中检索和关联。

为什么需要：没有完整时间线，攻击、误操作和故障都难以定位。

怎么工作：各数据源按统一时间发送到受保护的集中平台，限制删除和篡改，监控采集断流，并按适用要求留存。《网络安全法》第二十三条明确要求网络日志留存不少于六个月。

怎么用：先建立“应该接入的数据源清单”，再计算实际接入率和新鲜度；只看日志平台容量没有意义。

坏了怎么查：关键设备从未上报、字段缺少账号和源 IP、主机时间漂移、采集断流无告警、普通管理员能删除审计日志。

### 数据保护和备份恢复

是什么：根据数据重要性和敏感性控制访问、传输、存储、备份、恢复和销毁。

为什么需要：副本、日志、测试数据和导出文件同样可能泄露或被篡改。

怎么工作：数据分类分级、最小权限、加密、脱敏、备份、异地保护、恢复演练和介质销毁组成生命周期。

怎么用：每个关键数据集要有责任人、恢复点目标 RPO、恢复时间目标 RTO、备份位置和最近恢复结果。

坏了怎么查：备份任务全绿但多年未恢复、密钥和密文放在一起、生产数据直接复制到测试、离职人员仍可下载备份。

### 事件响应

是什么：发现安全事件后按分级、报告、遏制、清除、恢复和复盘流程行动。

为什么需要：有设备告警却没人判断和升级，控制仍然无效。

怎么工作：监测规则触发事件，值班人员确认影响，按预案召集角色并保留证据，恢复后形成整改项。

怎么用：把应急预案变成联系人、判定条件、命令、审批、证据和升级路径明确的 Runbook。

坏了怎么查：预案联系人已离职、演练只读稿、告警没有工单、处置过程中覆盖了原始证据。

## AIOps 平台自身的等保关注点

AIOps 平台经常具有“看得多、权限大、自动化强”的特点，应重点关注：

| 风险面 | 要检查什么 |
|---|---|
| 数据接入 | 是否采集个人信息、重要数据、凭证、生产日志和数据库内容，是否最小化 |
| 平台账号 | 管理员是否唯一身份、MFA、最小权限、定期复核和操作审计 |
| Agent / Collector | 安装权限、升级来源、证书、出站地址、缓存数据和失联检测 |
| 自动化 Runbook | 参数校验、审批、命令白名单、幂等、回滚、超时和审计 |
| LLM / RAG | 敏感数据外发、权限过滤、提示词注入、错误建议和高风险动作边界 |
| 知识库 | 文档权限、过期规则、版本、来源和删除机制 |
| 多租户 | 租户数据、检索、告警和管理员权限是否隔离 |
| 高可用 | 监控平台故障时是否仍能发现关键安全事件，数据是否丢失 |

LLM 可以帮助归类证据、摘要问题和推荐检查步骤，但不能替代正式定级、测评判断或责任人审批，更不能直接无审批执行封禁、删除、改权限等高风险动作。

## 用 AIOps 做持续合规

等保控制可以转成持续观测的数据流：

```text
CMDB / 云 API / Kubernetes API / IAM
  + 漏洞扫描 / EDR / 防火墙 / WAF
  + 主机、数据库、应用和审计日志
  + 备份、恢复、演练和变更工单
  -> 统一资产与控制标识
  -> 计算覆盖率、时效、漂移和超期风险
  -> 告警与整改工单
  -> 人工审批或安全 Runbook
  -> 复测并保存不可抵赖的证据
  -> 看板、测评材料和管理复盘
```

### 建议指标

| 指标 | 计算思路 | 异常说明 |
|---|---|---|
| 资产纳管率 | 已确认资产 / 发现资产 | 低说明存在未知或无人负责资产 |
| 日志接入覆盖率 | 正常上报的必接数据源 / 应接数据源 | 低说明审计时间线有盲区 |
| 特权账号 MFA 覆盖率 | 已启用 MFA 的特权账号 / 特权账号 | 低说明高权限身份保护不足 |
| 严重漏洞超期数 | 超过修复 SLA 的严重漏洞 | 持续上升说明整改流程失效 |
| 安全基线漂移数 | 当前配置偏离批准基线的项目 | 可能是未审批变更或配置回退 |
| 备份恢复成功率 | 验证恢复成功的关键对象 / 应演练对象 | 只看备份任务会高估恢复能力 |
| 证据新鲜度 | 当前时间 - 最近验证时间 | 证据过旧不能证明当前状态 |
| 高风险问题未闭环数 | 未完成整改或批准例外的高风险问题 | 应按业务影响升级 |

阈值不能照抄。先定义对象、责任、数据源、统计周期和例外规则，再根据等级、行业要求和风险承受能力设定。

## 控制台账配置

下面是学习用 YAML，不是官方备案或测评模板：

```yaml
system:
  id: aiops-demo                         # 保护对象的稳定标识，不使用容易变化的项目昵称
  name: AIOps 演示平台
  proposed_level: 3                     # 仅为实验假设，正式等级必须经过定级流程
  owner: platform-team
  boundary: demo-only                   # 指向边界说明和拓扑版本

controls:
  - id: IAM-01
    name: 特权账号唯一身份和多因素认证
    owner: iam-team
    status: partial                     # planned / partial / effective / exception
    evidence:
      - evidence/identity/admin-review-sanitized.csv
    last_verified: 2026-07-14
    next_review: 2026-08-14
    issue: 两个历史管理员账号未完成责任人确认
    ticket: SEC-101
```

| 字段 | 用途 | 常见坑 |
|---|---|---|
| `system.id` | 把资产、控制、证据和问题关联到同一对象 | 系统改名后历史证据断链 |
| `proposed_level` | 记录学习或初步定级结论 | 把自填数字误当成正式等级 |
| `control.id` | 给控制一个稳定编号 | 直接用产品名，换产品后无法追踪 |
| `owner` | 明确谁对控制有效性负责 | 只写“安全部”，没有具体角色 |
| `status` | 表示控制当前成熟状态 | 所有项目默认写 effective |
| `evidence` | 指向能复核的证据 | 只有截图，没有对象、时间和来源 |
| `last_verified` | 最近一次确认控制有效的时间 | 配置变化后仍沿用旧日期 |
| `next_review` | 下一次复核期限 | 没有到期提醒，证据长期失效 |
| `ticket` | 关联整改、例外或变更 | 问题与执行流程脱节 |

## 常用只读取证命令

这些命令只能提供局部证据，不能单独证明“通过等保”。执行前确认授权，输出中可能含主机名、IP、账号和安全配置，不要直接公开。

### Windows

```powershell
Get-ComputerInfo | Select-Object WindowsProductName, WindowsVersion, OsBuildNumber # 记录系统版本和补丁判断所需基础信息
Get-NetTCPConnection -State Listen | Sort-Object LocalPort                         # 查看监听端口，核对暴露面
Get-LocalUser | Select-Object Name, Enabled, LastLogon                            # 盘点本地账号状态和最近登录
Get-MpComputerStatus | Select-Object AntivirusEnabled, RealTimeProtectionEnabled  # 查看 Defender 防护状态
Get-WinEvent -FilterHashtable @{LogName='Security'; StartTime=(Get-Date).AddHours(-1)} -MaxEvents 20 # 抽查最近安全日志
w32tm /query /status                                                               # 检查时间同步来源和状态
```

正常结果：命令能返回当前主机数据，资产、端口和账号可以关联到责任人；安全日志有持续记录；时间源符合组织设计。`Access is denied` 通常表示当前终端没有相应读取权限，应走授权流程，不要为了取证关闭安全控制。

### Linux

```bash
hostnamectl                         # 记录主机标识、操作系统和内核版本
ss -lntup                           # 查看 TCP/UDP 监听端口及关联进程，部分进程信息需要授权
getent passwd                       # 查看账号目录，后续还要区分人员账号和服务账号
last -n 20                          # 抽查最近登录记录
systemctl is-enabled auditd         # 检查审计服务是否设置为开机启用
journalctl --since '1 hour ago'     # 抽查最近系统日志是否持续产生
timedatectl status                  # 检查时区和时间同步状态
```

正常结果：主机版本、监听端口、账号和日志都能被盘点，并能解释哪些是批准配置。命令失败时先检查发行版、服务名称和读取权限；最小化系统可能没有 `auditd`，这代表需要确认替代审计方案，不是把检查项直接删除。

### Kubernetes

```bash
kubectl get nodes -o wide                         # 盘点节点、版本和地址
kubectl get namespaces                            # 盘点逻辑租户或业务边界
kubectl get networkpolicy -A                      # 检查各 namespace 是否配置网络访问控制
kubectl auth can-i --list                         # 查看当前身份在当前上下文拥有的权限
kubectl get pods -A -o custom-columns='NAMESPACE:.metadata.namespace,NAME:.metadata.name,SERVICEACCOUNT:.spec.serviceAccountName' # 盘点工作负载使用的服务账号
```

正常结果：命令只返回当前身份被授权查看的资源。执行前先用 `kubectl config current-context` 确认集群，避免在错误环境取证；不要为了“看全”临时绑定 `cluster-admin`。

## 命令与证据字典

| 名称 | 作用 | 关键结果 | AIOps 场景 | 常见坑 |
|---|---|---|---|---|
| `Get-NetTCPConnection` / `ss` | 发现监听端口 | 地址、端口、进程 | 发现未登记暴露面 | 快照看不到历史短时端口 |
| `Get-LocalUser` / `getent passwd` | 盘点本地账号 | 账号、状态、登录信息 | 发现孤儿和长期未用账号 | 不能覆盖云 IAM、LDAP、数据库账号 |
| `Get-WinEvent` / `journalctl` | 抽查本地日志 | 时间、来源、事件 | 验证日志产生与时间线 | 本地有日志不等于集中接入和防篡改 |
| `w32tm` / `timedatectl` | 检查时间同步 | 时间源、同步状态 | 保证跨系统事件可关联 | 只看时区，不看实际偏差和时间源 |
| `Get-MpComputerStatus` | 查看 Defender 状态 | 实时防护、签名状态 | 终端防护覆盖监控 | 第三方 EDR 环境不能只用此命令判断 |
| `systemctl is-enabled auditd` | 检查审计服务自启 | enabled / disabled | 审计基线漂移告警 | enabled 不代表规则正确或日志已上送 |
| `kubectl auth can-i --list` | 查看当前 Kubernetes 权限 | verbs、resources、scope | RBAC 权限复核 | 只代表当前身份和上下文 |
| `kubectl get networkpolicy -A` | 盘点网络策略对象 | namespace、策略名 | 网络隔离覆盖率 | 有策略不代表默认拒绝或实际生效 |

## 入门实验：做一份可复核的安全基线证据包

### 实验目标

以自己的 Windows 学习机为对象，完成一次只读盘点，生成带时间、原始输出和哈希清单的证据包，再写出至少三个发现。这个实验不代表正式定级或测评。

### 实验步骤

1. 创建前面的 `labs/mlps-baseline` 目录。
2. 打开普通 PowerShell，执行下面脚本。
3. 查看输出，识别正常项、待确认项和风险项。
4. 在 `reports/self-check.md` 写发现、证据、风险、责任人和下一步。

```powershell
$Lab = 'labs/mlps-baseline'
$Stamp = Get-Date -Format 'yyyyMMdd-HHmmss'                                      # 用统一时间标记本次采集批次
$Out = "$Lab/evidence/system/$Stamp"
New-Item -ItemType Directory -Force $Out | Out-Null                              # 每次采集使用独立目录，避免覆盖历史

Get-Date -Format o | Out-File "$Out/collected-at.txt" -Encoding utf8            # 保存带时区的采集时间
Get-ComputerInfo | Select-Object WindowsProductName, WindowsVersion, OsBuildNumber |
  Out-File "$Out/os-version.txt" -Encoding utf8                                 # 保存操作系统版本
Get-NetTCPConnection -State Listen | Sort-Object LocalAddress, LocalPort |
  Format-Table -AutoSize | Out-File "$Out/listening-ports.txt" -Encoding utf8    # 保存监听端口快照
Get-LocalUser | Select-Object Name, Enabled, LastLogon |
  Format-Table -AutoSize | Out-File "$Out/local-users.txt" -Encoding utf8        # 保存本地账号快照
w32tm /query /status | Out-File "$Out/time-sync.txt" -Encoding utf8              # 保存时间同步状态

Get-ChildItem $Out -File | Get-FileHash -Algorithm SHA256 |
  Select-Object Path, Hash | Export-Csv "$Out/sha256-manifest.csv" -NoTypeInformation -Encoding utf8 # 为证据生成完整性清单
```

### 预期结果

新目录中至少出现：

```text
collected-at.txt
os-version.txt
listening-ports.txt
local-users.txt
time-sync.txt
sha256-manifest.csv
```

`sha256-manifest.csv` 中每个证据文件都有哈希值。`self-check.md` 至少回答：

- 哪些端口是预期的，谁负责？
- 是否有不认识、禁用但未说明、长期未使用的账号？
- 时间同步是否正常？
- 哪些结论只靠当前证据还不能判断？

### 验证方法

重新运行 `Get-FileHash` 并和清单比较。文件未变化时哈希应一致；修改任意证据文件后哈希会变化，这说明证据完整性需要重新确认。

### 如果没有成功

按顺序检查：

1. 当前目录是否允许创建 `labs/mlps-baseline`。
2. PowerShell 是否为 Windows PowerShell 5.1 或 PowerShell 7。
3. 某条命令是否因为系统组件不存在或权限不足失败。
4. 失败命令是否留下空文件，报告里要如实记录“未采集成功”。
5. 路径中是否包含真实敏感信息，提交 GitHub 前必须脱敏。

## 有效证据应该包含什么

一张截图通常不够。高质量证据至少能回答：

| 问题 | 说明 |
|---|---|
| 证明哪个对象 | 系统、集群、账号、设备或控制的唯一标识 |
| 证明什么控制 | 对应的控制目标和检查结论 |
| 谁采集 | 采集人、账号或自动化任务身份 |
| 什么时候采集 | 带时区时间和证据有效期 |
| 从哪里采集 | 命令、API、页面、日志源和环境 |
| 原始结果是什么 | 未被摘要隐藏的必要字段 |
| 如何防篡改 | 访问控制、哈希、签名、只读存储或审计日志 |
| 发现问题后做了什么 | 工单、变更、复测、例外审批和到期时间 |

自动采集可以减少重复劳动，但要保护采集账号和证据仓库。自动化任务失败或数据源断流也必须告警，否则“没有发现问题”可能只是“没有采到数据”。

## 测评前准备清单

### 范围与架构

- 保护对象名称、责任主体、等级和备案材料一致。
- 最新拓扑包含公网入口、云服务、第三方接口、远程运维、管理面和数据流。
- 资产清单覆盖网络、主机、数据库、中间件、容器、账号、证书和关键数据。
- 云服务共享责任矩阵明确，云平台报告在有效期且范围匹配。

### 技术控制

- 特权账号唯一身份、MFA、审批、复核和离职回收有证据。
- 网络边界规则有业务依据、责任人、期限和变更记录。
- 主机、数据库、应用和容器基线与当前生产一致。
- 漏洞扫描范围完整，严重问题已整改、复测或批准例外。
- 日志数据源、字段、时间同步、留存、防篡改和查询能力可验证。
- 备份覆盖关键对象，并有近期恢复演练记录。

### 管理控制

- 制度版本、审批、发布、培训和定期评审记录完整。
- 组织职责和实际人员一致，关键岗位没有无人负责。
- 开发、测试、上线、变更、供应商和外部人员流程有真实工单。
- 应急预案、联系人、演练、问题和改进项形成闭环。

### 测试安全

- 明确测试窗口、目标 IP、账号、工具、流量限制、停止条件和联系人。
- 生产测试经过授权，不使用会破坏数据或影响业务的默认参数。
- 测评证据安全传输、最小披露，并约定保存和销毁方式。

## 常见问题与排障

### 测评范围一直对不上

可能原因：系统边界、CMDB、云账号、拓扑和备案名称各自维护。

检查方法：以业务为主线，从域名和入口向下追到服务、集群、数据库、数据和外部依赖，再与资产和备案材料逐项对账。

解决思路：建立稳定 `system_id`，所有资产、证据、变更和问题都关联它；重大边界变化触发重新评估。

### 日志平台有数据，但仍被认为不充分

可能原因：关键数据源未接、字段不够、时间不同步、日志可被删除、留存不足、查询不到管理操作，或采集断流没有告警。

检查方法：从“应接清单”随机抽取数据源，验证一条真实操作能否在规定时间内被检索并关联到个人、来源、目标和结果。

解决思路：治理覆盖率、字段标准、时间同步、权限、防篡改、留存和断流告警，而不是只扩日志磁盘。

### 漏洞已经扫描，问题却没有闭环

可能原因：扫描范围不全、结果没有资产责任人、只关闭工单未复测、业务例外没有到期时间。

检查方法：从一个严重漏洞反查资产、版本、责任人、变更和复测；再从 CMDB 抽资产确认是否被扫描。

解决思路：把漏洞工单和资产、版本、SLA、补偿控制、复测证据绑定。

### 有备份，但恢复能力无法证明

可能原因：只监控备份任务成功，没有测试介质、密钥、依赖、恢复步骤和业务验证。

检查方法：随机选择关键对象，在隔离环境按 Runbook 恢复，记录实际 RPO、RTO 和数据校验结果。

解决思路：按业务重要性制定演练计划，把恢复结果而不是备份成功率作为核心指标。

### 云厂商有报告，租户仍有很多问题

可能原因：云平台报告覆盖基础设施，但租户安全组、IAM、操作系统、Kubernetes、应用、数据和日志配置不在其责任范围。

检查方法：逐项查看报告范围与共享责任矩阵，标出云厂商、租户和共同控制。

解决思路：复用可复用证据，同时补齐租户侧控制，不重复测评也不扩大推断。

### 制度齐全，现场执行对不上

可能原因：制度来自模板，没有结合组织、工具和真实流程。

检查方法：从制度随机选一个动作，追踪最近一次工单、审批、操作日志和复核结果。

解决思路：删除无法执行的空话，把责任、触发、步骤、工具、证据和升级写清楚。

### 测评后发布新版本，证据很快失效

可能原因：测评和变更管理分离，新增接口、资源、权限或数据没有触发安全评审。

检查方法：对比测评时和当前的域名、云资源、镜像、依赖、权限、网络策略和数据流。

解决思路：把控制检查放进 CI/CD 和变更门禁，重大变化触发差距分析、证据重采和必要的定级备案评估。

## 持续运营责任矩阵

| 角色 | 主要责任 |
|---|---|
| 业务负责人 | 确认业务重要性、影响、停机和数据风险，接受残余风险 |
| 网络运营者 / 建设使用单位 | 对等级保护全过程和持续安全承担主体责任 |
| 安全团队 | 解释控制、组织差距分析、监督整改、事件响应和证据治理 |
| 运维 / SRE / AIOps | 维护资产、配置、日志、备份、监控、变更和自动化证据 |
| 开发团队 | 处理应用身份、权限、接口、依赖、漏洞和安全开发问题 |
| 云服务商 / 供应商 | 按合同和责任边界提供服务、控制与证据 |
| 测评机构 | 按适用标准独立开展等级测评并出具报告 |
| 公安机关和行业主管部门 | 按职责受理备案、指导、监督和检查 |

外包可以转移执行工作，通常不能把主体责任一起外包。合同应写明权限、日志、事件报告、漏洞、证据、退出和数据销毁要求。

## 什么时候要重新评估

出现下面变化时，不要等下一次测评才处理：

- 核心业务功能、服务对象或影响范围明显变化。
- 合并或拆分系统，新增公网入口、移动端、物联网或工业控制场景。
- 从自建机房迁移到云，或在 IaaS、PaaS、SaaS 之间改变模式。
- 新增重要数据、个人信息处理、跨境或大规模数据汇聚。
- 更换关键网络架构、身份系统、数据库、Kubernetes 集群或安全管理平台。
- 发生重大安全事件，暴露原等级或控制强度可能不足。
- 法律、标准、行业和属地要求变化。

评估结论可能是更新证据、补做差距分析、修改控制、重新定级或重新备案。具体动作应和主管部门、备案机关及测评机构确认。

## 面试怎么讲

网络安全等级保护是我国网络安全的基础制度，核心不是买安全产品或拿一次测评报告，而是先确定具体保护对象，根据系统被破坏后影响的对象和程度定级，再完成备案、安全建设整改、等级测评和持续监督运营。

技术侧我会按安全物理环境、安全通信网络、安全区域边界、安全计算环境和安全管理中心检查；管理侧看制度、机构、人员、建设和运维。落地时，我会把每个控制关联到资产、责任人、配置、日志、工单和证据，并用 AIOps 持续监控资产纳管率、日志覆盖率、特权账号 MFA、漏洞超期、基线漂移和恢复演练。云平台通过测评不能替代租户业务自身责任，等保也不能替代数据安全、个人信息保护、关基保护或密码应用等其他合规要求。
