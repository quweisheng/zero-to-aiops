# 技术栈精讲写作标准

这个标准用来约束本仓库后续所有“一技术一文件”的文章。目标是写成小白能照着学、面试能拿来复盘、项目能直接引用的 AIOps 学习笔记。

## 参考站扫描结论

参考：[二哥的 Java 进阶之路：一文让你彻底了解 Java](https://javabetter.cn/overview/what-is-java.html)。

这个站点值得借鉴的不是具体人设和段子，而是它的讲解方法：

1. 先让读者进入一个真实的学习场景，再开始讲概念。
2. 用对话推进节奏，让新手把“不懂、疑惑、追问”说出来。
3. 每个术语第一次出现时，都用大白话拆一遍。
4. 先讲为什么，再讲是什么，最后讲怎么做。
5. 用历史、类比、截图、代码、命令和小实验交替推进，避免连续堆概念。
6. 一段只解决一个小问题，读者跟得上再进入下一段。
7. 结尾要落到应用场景、就业价值、学习路线或下一步行动。

本仓库要借鉴这种“有人带着学”的感觉，但不能照搬它的角色、原句、笑点和图片。AIOps 技术栈的主线应该是：

```text
运维真实问题
  -> 小白疑问
  -> 大白话解释
  -> 核心机制
  -> 最小实验
  -> 常见故障
  -> AIOps 场景
  -> 面试表达
  -> GitHub 学习证据
```

逐行解释：

| 行 | 内容 | 说明 |
|---|---|---|
| 第 2 行 | <code>  -&gt; 小白疑问</code> | 这一行表示上一级主题下的子项“小白疑问”。`->` 只是知识地图里的层级符号，真正要理解的是这句话里的操作或概念。 |
| 第 3 行 | <code>  -&gt; 大白话解释</code> | 这一行表示上一级主题下的子项“大白话解释”。`->` 只是知识地图里的层级符号，真正要理解的是这句话里的操作或概念。 |
| 第 4 行 | <code>  -&gt; 核心机制</code> | 这一行表示上一级主题下的子项“核心机制”。`->` 只是知识地图里的层级符号，真正要理解的是这句话里的操作或概念。 |
| 第 5 行 | <code>  -&gt; 最小实验</code> | 这一行表示上一级主题下的子项“最小实验”。`->` 只是知识地图里的层级符号，真正要理解的是这句话里的操作或概念。 |
| 第 6 行 | <code>  -&gt; 常见故障</code> | 这一行表示上一级主题下的子项“常见故障”。`->` 只是知识地图里的层级符号，真正要理解的是这句话里的操作或概念。 |
| 第 7 行 | <code>  -&gt; AIOps 场景</code> | 这一行要理解这些英文词：`AIOps` 是智能运维。`->` 只是知识地图里的层级符号，不是要学习的概念。 |
| 第 8 行 | <code>  -&gt; 面试表达</code> | 这一行表示上一级主题下的子项“面试表达”。`->` 只是知识地图里的层级符号，真正要理解的是这句话里的操作或概念。 |
| 第 9 行 | <code>  -&gt; GitHub 学习证据</code> | 这一行要理解这些英文词：`GitHub` 是代码托管平台。`->` 只是知识地图里的层级符号，不是要学习的概念。 |

如果文本图、目录树或知识地图里出现 `-> Tutorial`、`-> connect`、`service -> queue` 这类写法，逐行解释不能写成“箭头表示从左到右流动”。真正要解释的是这一行里的英文词、缩写或组件名，例如 `Tutorial` 是教程章节，`connect` 是连接，`queue` 是消息队列。没有英文的中文行，则解释这个子主题或操作本身。

如果示例是普通文本里的英文键值对，也不能写成“文本示例行”。要拆开解释字段和值，例如 `service=order-api` 要写清楚：`service` 是服务名称字段，`order-api` 是具体服务名；`severity=critical` 要写清楚：`severity` 是告警严重级别字段，`critical` 表示严重级别。纯中文文本行已经能直接读懂，不需要为了凑表格再额外解释。

## 文章不是资料搬运

每篇文章都必须基于官方文档和实际学习场景重新组织。可以引用官方链接，可以引用少量配置片段，但不能复制官方全文，也不能把官网内容逐段翻译。

写作时优先回答三个问题：

1. 这个技术为什么和 AIOps 有关？
2. 一个运维小白学它，第一天应该掌握什么？
3. 学完以后能留下什么证据放到 GitHub？

## 固定结构

每个技术栈文件至少包含这些部分：

| 部分 | 目的 |
|---|---|
| 学习目标 | 告诉读者学完能做什么 |
| 官方资料 | 给出官网入口，避免资料来源混乱 |
| 场景开场 | 用一个运维或学习场景把读者带进来 |
| 一句话人话版 | 先用一句白话讲清楚它大概是什么 |
| 小白追问 | 主动写出新手最可能卡住的问题 |
| 为什么要学 | 连接运维、SRE、AIOps 和求职 |
| 是什么 | 用大白话讲定义 |
| 解决什么问题 | 解释它出现的原因 |
| 核心原理 | 讲机制，不只讲命令 |
| 架构和数据流 | 说明组件关系 |
| 安装与启动 | 让读者能跑起来 |
| 配置详解 | 解释配置项为什么这么写 |
| 常用命令或查询 | 给出入门必会命令 |
| AIOps 场景 | 说明它在指标、日志、告警、自动化、AI 中的位置 |
| 入门实验 | 给出可复现的小实验 |
| 常见故障排查 | 写现象、原因、检查、解决 |
| 学习检查清单 | 帮读者确认自己学会了什么 |
| 面试题 | 把知识转成表达能力 |
| 学习证据 | 指导读者把成果提交到 GitHub |

## 通俗讲解流程

每篇文章尽量按下面的节奏展开。

1. 场景开场：不要一上来定义术语。先写一个读者可能遇到的场景，比如“半夜告警响了，只知道 CPU 高，但不知道哪个进程惹的事”。
2. 一句话人话版：用一句不超过 80 字的话解释这个技术。
3. 新手追问：写出 2 到 4 个读者会问的问题，例如“为什么不能直接看日志？”“为什么容器里 localhost 不一样？”。
4. 术语拆解：第一次出现英文缩写、组件名、配置字段时，立刻解释它的作用。
5. 机制图解：能用文字图就画文字图，先把数据从哪里来到哪里去讲清楚。
6. 最小实验：给一条最短路径，让读者今天就能跑起来。
7. 排障回路：每个实验后面必须写“如果没成功，先看哪里”。
8. AIOps 落点：最后说明它能给异常检测、告警降噪、根因分析、自动化修复提供什么。
9. 面试表达：把知识点改写成可以开口说的回答。
10. 学习证据：明确这篇学完后应该提交什么文件、截图或笔记。

## 讲解颗粒度

不要只写：

```text
Prometheus 是监控系统，可以采集指标。
```

逐行解释：

| 行 | 内容 | 说明 |
|---|---|---|
| 第 1 行 | <code>Prometheus 是监控系统，可以采集指标。</code> | 这一行里的英文要这样读：`Prometheus` 是指标采集和告警规则评估系统。 |

要写成：

```text
Prometheus 负责周期性访问目标服务的 /metrics 接口，把返回的数值样本保存为时间序列。每条时间序列由指标名和标签唯一确定，所以同一个 http_requests_total 指标可以按 method、status、instance 区分不同维度。
```

逐行解释：

| 行 | 内容 | 说明 |
|---|---|---|
| 第 1 行 | <code>Prometheus 负责周期性访问目标服务的 /metrics 接口，把返回的数值样本保存为时间序列。每条时间序列由指标名和标签唯一确定，所以同一个 http_requests_total 指标可以按 method、status、instance 区分不同维度。</code> | `Prometheus 负责周期性访问目标服务的 /metrics 接口，把返回的数值样本保存为时间序列。每条时间序列由指标名和标签唯一确定，所以同一个 http_requests_total 指标可以按 method、status、instance 区分不同维度。` 是文件、目录、接口路径或匹配模式示例，用来告诉读者真实项目里应该把学习证据、配置或代码放在哪里。 |

讲解要覆盖“定义、为什么、怎么工作、怎么配置、怎么验证、坏了怎么查”。

## 深讲硬性标准

“通俗”不是“浅”。后续每篇真正的精讲文章，必须同时满足下面这些要求，否则只能算入门提纲，不能算精讲。

### 1. 先给官方知识地图

文章要先说明这门技术的官方资料结构，告诉读者官方文档大概分成哪些部分，以及本篇会覆盖哪些部分。

例如 Linux 不能只写“会用 top、free、df”。它至少要讲清：

- Linux 内核是什么。
- 用户态和内核态怎么分工。
- 系统调用是什么。
- 进程、线程、调度、内存、文件系统、网络、设备、权限分别是什么。
- `/proc`、`/sys`、`/etc`、`/var/log` 这些目录为什么重要。
- 常用命令来自哪些用户态工具包，而不是都属于 Linux 内核。

### 2. 每个知识点都要按“五件套”讲

每个核心概念必须包含：

| 项 | 要求 |
|---|---|
| 是什么 | 一句话定义，不绕 |
| 为什么需要 | 它解决什么问题 |
| 怎么工作 | 数据流、调用链或状态变化 |
| 怎么看 / 怎么用 | 对应命令、配置、API 或页面 |
| 坏了怎么查 | 常见现象、检查顺序、修复思路 |

### 3. 命令必须像字典一样讲

命令不能只列名字。至少要写：

| 项 | 示例 |
|---|---|
| 命令 | `ps` |
| 作用 | 查看进程快照 |
| 常用语法 | `ps -ef`、`ps aux` |
| 关键字段 | PID、PPID、USER、STAT、CMD |
| 示例 | `ps -ef | grep nginx` |
| 预期输出 | 能看到进程号和启动命令 |
| AIOps 场景 | 告警后确认服务进程是否存在 |
| 常见坑 | `grep` 自己也会出现在结果里 |

### 4. 示例代码必须逐行注释

凡是技术栈文章里的示例块，不管是代码、命令、SQL、PromQL、LogQL、YAML、JSON、配置文件、伪代码还是终端输出，都要让真正的小白能逐行看懂。

硬性要求：

- 每一行都要有中文注释，或者在代码块下面紧跟逐行解释表。
- 注释要说明“这一行做什么、为什么要写、真实 AIOps 场景中哪里需要改”。
- 不能只在代码块前写一句“下面是示例”，然后留一整块没有解释的内容。
- 如果某种语法不适合写行内注释，比如 JSON，就在代码块下面用“逐行解释”表格解释每个字段。
- 新增或改写任何 `docs/tech-stack/**/*.md` 文件时，要顺手检查这篇文章里的示例块，未解释的示例块要补齐。

SQL 示例应该这样写：

```sql
SELECT service_name, COUNT(*) AS alert_count -- 取服务名，并统计每个服务有多少条告警，统计结果命名为 alert_count
FROM alerts                                  -- 从 alerts 告警表里读取数据
WHERE created_at >= NOW() - INTERVAL 1 DAY   -- 只看最近 1 天产生的告警，避免把历史旧告警也算进去
GROUP BY service_name                        -- 按服务名分组，让每个服务单独统计
ORDER BY alert_count DESC                    -- 按告警数量从多到少排序，先看到最吵的服务
LIMIT 10;                                    -- 只返回前 10 个服务，适合做告警治理 TopN
```

逐行解释：

| 行 | 内容 | 说明 |
|---|---|---|
| 第 1 行 | <code>SELECT service_name, COUNT(*) AS alert_count -- 取服务名，并统计每个服务有多少条告警，统计结果命名为 alert_count</code> | 选择最终要返回的字段或计算结果，是查询结果表头的来源。 |
| 第 2 行 | <code>FROM alerts                                  -- 从 alerts 告警表里读取数据</code> | 指定从哪张表读取数据，是 SQL 逻辑执行的起点。 |
| 第 3 行 | <code>WHERE created_at &gt;= NOW() - INTERVAL 1 DAY   -- 只看最近 1 天产生的告警，避免把历史旧告警也算进去</code> | 过滤原始数据行，只保留符合条件的记录。 |
| 第 4 行 | <code>GROUP BY service_name                        -- 按服务名分组，让每个服务单独统计</code> | 按指定字段分组，让每组单独统计或聚合。 |
| 第 5 行 | <code>ORDER BY alert_count DESC                    -- 按告警数量从多到少排序，先看到最吵的服务</code> | 对查询结果排序，让最重要或最新的数据排在前面。 |
| 第 6 行 | <code>LIMIT 10;                                    -- 只返回前 10 个服务，适合做告警治理 TopN</code> | 限制返回行数，避免结果太多影响阅读或性能。 |

YAML 示例应该这样写：

```yaml
scrape_interval: 15s  # Prometheus 每 15 秒抓一次指标；太短会增加压力，太长会让告警变慢
```

逐行解释：

| 行 | 内容 | 说明 |
|---|---|---|
| 第 1 行 | <code>scrape_interval: 15s  # Prometheus 每 15 秒抓一次指标；太短会增加压力，太长会让告警变慢</code> | `scrape_interval` 是Prometheus 抓取指标的时间间隔字段，`15s  # Prometheus 每 15 秒抓一次指标；太短会增加压力，太长会让告警变慢` 是示例取值，真实 AIOps 场景里要换成自己的服务、环境、路径或阈值；真实 AIOps 环境里要按自己的服务名、端口、路径或策略调整。 |

Bash 示例应该这样写：

```bash
docker ps # 查看正在运行的容器，确认实验服务是不是已经启动
```

逐行解释：

| 行 | 内容 | 说明 |
|---|---|---|
| 第 1 行 | <code>docker ps # 查看正在运行的容器，确认实验服务是不是已经启动</code> | 执行 Docker 容器命令，用来启动、查看、停止或构建容器化实验环境。 |

JSON 示例不能写注释时，要在下面补表格：

```json
{
  "service": "order-api",
  "severity": "critical"
}
```

逐行解释：

| 行 | 内容 | 说明 |
|---|---|---|
| 第 1 行 | <code>{</code> | 对象开始，表示下面是一组键值对配置。 |
| 第 2 行 | <code>  "service": "order-api",</code> | `service` 是服务名称字段，`order-api` 是具体服务名，表示这条记录属于这个服务；真实环境要根据自己的告警、服务或接口返回调整。 |
| 第 3 行 | <code>  "severity": "critical"</code> | `severity` 是告警严重级别字段，`critical` 表示严重级别，通常表示需要优先处理；真实环境要根据自己的告警、服务或接口返回调整。 |
| 第 4 行 | <code>}</code> | 对象结束，表示这一组键值对配置到这里结束。 |

| 行 | 含义 |
|---|---|

| `"severity": "critical"` | 告警级别是严重，通常需要更快响应或人工确认。 |

| 第 3 行 | <code>  -&gt; 再学最小命令</code> | 这一行表示上一级主题下的子项“再学最小命令”。`->` 只是知识地图里的层级符号，真正要理解的是这句话里的操作或概念。 |
| 第 4 行 | <code>  -&gt; 再做实验</code> | 这一行表示上一级主题下的子项“再做实验”。`->` 只是知识地图里的层级符号，真正要理解的是这句话里的操作或概念。 |

### 6. 明确边界和取舍

如果某个领域本身没有“全部命令”这种单一清单，文章要说清楚边界。

例如 Linux 命令来自 GNU coreutils、util-linux、procps-ng、iproute2、systemd、发行版包管理器等不同项目。文章可以定义“AIOps 入门必会命令全集”，但不能假装 Linux 内核官方有一个包含所有用户态命令的清单。

## 小白友好规则

- 开头可以写 2 到 5 行轻量对话，但必须服务于概念，不要为了热闹而热闹。
- 每 3 到 5 个自然段要给读者一个停靠点：小结、文字图、示例、命令、表格或检查问题。
- 第一次出现英文缩写时，要解释全称和作用。
- 第一次出现配置字段时，要解释字段含义。
- 命令后面要写预期结果。
- 示例块里的每一行都要有注释或逐行解释，尤其是 SQL、YAML、JSON、PromQL、LogQL、shell 和 Python。
- 不假设读者已经懂 Kubernetes、Python、机器学习。
- 能画数据流就用文字图说明。
- 少写“显而易见”“很简单”这类话。对新手来说，不简单才是正常的。

## 语言风格

- 像一个有经验的运维师傅带新人，而不是像考试资料在背定义。
- 可以口语化，但不要油腻，不要复制别人的梗。
- 段落要短，长概念拆成多段讲。
- 多用“你可以把它理解成”“这一步其实是在做”“坏了通常先看”。
- 少用空泛形容词，比如“强大”“高效”“灵活”。要写强在哪里、高效在哪里、灵活在哪里。
- 每篇都要有 AIOps 视角：这项技术如何进入指标、日志、链路、告警、自动化、模型或知识库。

## AIOps 关联规则

每篇文章必须明确它属于 AIOps 链路的哪一段：

- 数据采集：Prometheus、OpenTelemetry、Loki、Filebeat。
- 数据存储：Elasticsearch、ClickHouse、VictoriaMetrics、MySQL。
- 数据展示：Grafana。
- 告警治理：Alertmanager、Grafana Alerting。
- 自动化执行：Ansible、Runbook Automation、GitHub Actions。
- 智能分析：pandas、scikit-learn、LLM、RAG。
- 稳定性方法：SLI、SLO、RCA、事件响应、变更管理。

## 学习证据规则

每学完一个技术，至少提交一种证据：

- 配置文件：如 `prometheus.yml`、`compose.yaml`、dashboard JSON。
- 截图：如 targets 页面、Grafana dashboard、构建成功页面。
- 笔记：如“Counter 和 Gauge 的区别”。
- 排障记录：如“为什么容器里不能访问 localhost:9090”。
- 小项目：如“用 Prometheus + Grafana 监控一个 FastAPI demo”。
