# IT 项目管理 / PMP 深讲

> 学习目标：从零理解项目、产品、项目集和运营的边界，能为一个 AIOps 项目完成立项、范围拆分、进度与成本计划、风险和变更控制、敏捷或混合交付、上线验收和复盘；同时能按当前 PMP 考纲组织知识，并在面试中解释机制、取舍和事故处理。

## 学习边界与版本说明

本文讲的是 **IT 项目交付能力**，不是考试题库，也不会复述受版权保护的 PMI 教材或真实考试题。

- 知识体系以 PMI 在 2025 年 11 月发布的《PMBOK Guide》第八版为当前基线。它强调 6 项核心原则、7 个绩效域，以及价值、适应性和问责。

- 软件交付同时参考《Scrum Guide》、Agile Manifesto、The Kanban Guide 和 DORA 软件交付指标。
- PMP 是项目管理专业认证，PMBOK 是知识体系指南，二者都不等于真实项目经验。
- 读完本文不能保证通过考试或获得岗位。考试资格、培训、申请审核和备考应以 PMI 官网最新要求为准；技术项目岗位还需要 Linux、网络、编码、系统设计和生产经验。

## 官方知识地图

```text
战略与价值
  -> 商业论证 / 预期收益 / 成功标准
  -> 项目章程 / 治理 / 决策权限
  -> 干系人 / 团队 / 沟通
  -> 需求 / 范围 / WBS / 验收
  -> 进度 / 成本 / 资源 / 采购
  -> 质量 / 风险 / 问题 / 变更
  -> 预测型 / 敏捷 / 混合型交付
  -> 发布 / 运营就绪 / 收益确认
  -> 复盘 / 知识沉淀 / 持续改进
```

`WBS` 是 Work Breakdown Structure，工作分解结构；它把可交付成果逐层拆成可估算、可分派、可验收的工作包。项目管理的主线不是“填表”，而是让目标、承诺、执行证据和决策持续一致。

## 场景开场

公司计划用 12 周建设一套 AIOps 告警治理平台：

- 接入 Prometheus、日志平台和工单系统。
- 将重复告警减少 40%。
- 为 20 个核心服务建立告警负责人和 Runbook。
- 自动化动作只能在测试环境执行，生产环境必须审批。
- 上线日期已经对外承诺，但安全团队、业务团队和平台团队使用不同的排期。

如果项目经理只收集“完成百分比”，第 10 周很可能才发现：数据权限没批、验收口径不一致、生产回滚没演练、算法离线准确率也不能代表线上收益。

真正的项目管理要在问题变成事故之前，让这些依赖、假设、风险和决策变得可见。

## 一句话人话版

项目管理就是在资源有限、信息不完整和变化不断发生的情况下，组织人和证据，把一个有起止时间的目标交付成可验证的业务价值。

## 为什么 AIOps 工程师要学项目管理

AIOps 很少是一个人完成的脚本。它通常横跨监控、日志、网络、数据库、安全、算法、应用团队和生产变更流程。

项目管理能力能帮助你：

- 把“减少告警”改写成有基线、目标、时间窗和数据源的成功标准。
- 把平台、数据、模型、自动化和运营工作拆成可交付成果。
- 发现权限、接口、环境和人员等跨团队依赖。
- 在上线前完成容量、安全、回滚、监控和运维移交。
- 用风险和变更机制保护生产环境，而不是用加班掩盖不确定性。
- 把项目经历讲成“目标、约束、行动、证据、结果、复盘”的面试故事。

## 项目管理、PMP 与 PMBOK 是什么

### 项目管理

- **是什么**：运用知识、技能、工具和领导力，使项目产生预期成果。
- **为什么需要**：技术正确不代表范围、时间、成本、风险和价值同时可控。
- **怎么工作**：持续连接目标、计划、执行数据、风险、决策和反馈。
- **怎么用或观察**：看章程、交付物、看板、基线、风险台账、变更记录和验收证据是否一致。
- **坏了怎么查**：若状态长期“正常”却突然延期，先检查数据是否滞后、依赖是否隐藏、完成定义是否模糊。

### PMBOK

- **是什么**：Project Management Body of Knowledge，项目管理知识体系指南。
- **为什么需要**：为不同组织和行业提供共同语言与可裁剪的实践框架。
- **怎么工作**：用原则指导行为，用绩效域观察结果，再按项目环境裁剪方法。
- **怎么用或观察**：用它检查治理、范围、进度、财务、干系人、资源和风险是否存在明显空白。
- **坏了怎么查**：若团队机械套模板，应检查每份文档支持哪个决策；没有用途的文档不应继续制造。

### PMP

- **是什么**：Project Management Professional，PMI 的项目管理专业认证。
- **为什么需要**：验证申请者满足一定经验和培训条件，并通过以情境判断为主的考试。
- **怎么工作**：考试考查人员、过程和商业环境中的判断，不只是术语记忆。
- **怎么用或观察**：先用官方资格页面核对学历、项目领导经验和 35 小时培训要求，再按考试内容大纲准备。
- **坏了怎么查**：若模拟题正确率停滞，应按领域和任务分析误判原因，而不是背答案位置。

## 项目、项目集、项目组合、产品与运营

| 概念 | 关注点 | AIOps 示例 |
|---|---|---|
| 项目 Project | 有明确起止时间，交付独特成果 | 12 周上线告警治理平台 |
| 项目集 Program | 协调多个相关项目以获得单项目无法实现的收益 | 统一可观测平台、CMDB 治理、自动化处置三个项目 |
| 项目组合 Portfolio | 按战略、风险和资源选择投资 | 决定今年投资可观测性、容灾还是成本治理 |
| 产品 Product | 持续发现和满足用户需求 | 长期运营内部 AIOps 平台 |
| 运营 Operations | 持续、重复地提供服务 | 7×24 值守、容量巡检、告警处理 |

一个平台可以先作为项目建设，再作为产品持续演进，最后进入日常运营。项目结束不等于平台停止变化。

### 输出、成果、收益与价值

- **输出 Output**：做出来的东西，例如告警聚合服务。
- **成果 Outcome**：行为或能力发生变化，例如值班人员能在一个页面完成研判。
- **收益 Benefit**：可衡量的改善，例如重复告警下降 40%。
- **价值 Value**：收益与成本、风险、时间和战略意义的综合判断。

只按“页面是否上线”验收，会奖励输出而忽略价值。项目应同时定义技术验收和业务收益观察期。

## PMBOK 第八版的主干

PMBOK 第八版使用 6 项核心原则和 7 个绩效域。本文不逐句复述原文，而把它们转成实际检查问题。

### 六项原则怎样落地

| 原则方向 | 实际检查问题 |
|---|---|
| 整体视角 | 技术、流程、人员、供应商和运营之间有什么反馈和副作用？ |
| 聚焦价值 | 交付物怎样改变业务结果，收益由谁确认？ |
| 质量内建 | 质量是在开发中形成，还是上线前才集中测试？ |
| 负责任领导 | 谁拥有决策权，冲突怎样升级，错误是否可以透明暴露？ |
| 可持续性 | 方案是否透支人员、预算、环境或长期可维护性？ |
| 赋能团队 | 目标和边界是否清楚，团队能否在边界内自主决策？ |

### 七个绩效域怎样联动

治理、范围、进度、财务、干系人、资源和风险不是七张互不相干的表。例如新增“生产自动处置”会同时改变：

- 治理：谁能批准自动化权限。
- 范围：增加审批、审计和回滚能力。
- 进度：安全评审成为新依赖。
- 财务：可能增加环境、许可证和人力成本。
- 干系人：安全、内审和业务负责人需要参与。
- 资源：需要自动化与安全工程师。
- 风险：误操作的影响范围扩大。

项目经理要做的是跨域影响分析，而不是分别更新七份孤立文档。

## 五个过程组不是五个固定阶段

启动、规划、执行、监控与控制、收尾是观察管理活动的五个过程组。它们可以重叠，也会在迭代中反复出现。

```text
启动：确认为什么做、谁授权、成功是什么
规划：决定怎样做、需要什么、如何应对不确定性
执行：组织人员和资源产生交付物
监控与控制：比较计划与事实，处理偏差、风险和变更
收尾：验收、移交、结算、复盘并释放资源
```

错误理解是“先启动一个月，再规划一个月，之后绝不变化”。真实 IT 项目会在每个迭代重新规划局部范围，同时保持目标和治理边界稳定。

## 角色、责任与决策权

| 角色 | 核心责任 | 不应被混淆为 |
|---|---|---|
| Sponsor 发起人 | 授权项目、提供资源、处理重大障碍、确认价值 | 每天催任务的人 |
| Project Manager 项目经理 | 整合目标、人员、计划、风险、沟通和决策 | 所有技术决策的唯一负责人 |
| Product Owner 产品负责人 | 最大化产品价值，维护产品目标和待办项顺序 | 需求记录员 |
| Tech Lead 技术负责人 | 技术方案、工程质量、关键技术风险 | 项目全部商务责任人 |
| Scrum Master | 帮助团队正确应用 Scrum 并改进系统 | 传统意义上的任务分配者 |
| PMO 项目管理办公室 | 提供治理、标准、数据和组合级支持 | 只收周报的部门 |
| Operations 运营团队 | 运行、监控、响应、恢复和持续改进 | 上线最后一天才通知的接收方 |

### RACI

`RACI` 用于澄清一项工作中的责任：

- `R`，Responsible，实际执行。
- `A`，Accountable，对结果最终负责，通常每项只有一个。
- `C`，Consulted，决策前需要征询。
- `I`，Informed，需要被告知。

RACI 不能代替团队协作。若一项工作有五个 `A`，实际含义往往是没人最终负责。

## 生命周期与裁剪

### 预测型

范围较稳定，先详细规划再按阶段交付。适合硬件采购、机房建设、合规改造等变更成本高的工作。

### 迭代型

通过重复循环改善解决方案，例如多轮优化异常检测模型。

### 增量型

分批交付可用能力，例如先接入指标，再接入日志，最后接入自动化。

### 敏捷或适应型

需求不确定且反馈频繁，以短周期、可工作的增量和持续反馈降低风险。

### 混合型

整体里程碑、预算、合规和采购采用预测型，软件功能采用敏捷迭代。多数企业基础设施和 AIOps 项目更接近混合型。

### 裁剪不是随意删步骤

裁剪要回答：

1. 项目的不确定性和变更成本多高？
2. 监管、审计和安全要求多强？
3. 团队规模、分布和经验怎样？
4. 交付物能否频繁发布和回滚？
5. 干系人多久能提供一次有效反馈？
6. 哪些文档支持真实决策，哪些只是惯例？

## 从立项到价值实现

### 1. 商业论证与项目章程

#### 商业论证 Business Case

- **是什么**：说明为什么值得投资，包括问题、候选方案、成本、收益和风险。
- **为什么需要**：避免团队因为“技术很热门”而启动没有价值的问题。
- **怎么工作**：比较不做、最小改进和完整建设等选项。
- **怎么用或观察**：收益必须有基线、指标、负责人和观察时间窗。
- **坏了怎么查**：若目标频繁变化，检查发起人是否真正同意问题定义和成功标准。

#### 项目章程 Project Charter

章程是项目获得正式授权的轻量文件，至少包含：

| 字段 | 要回答的问题 |
|---|---|
| 背景与问题 | 当前损失或机会是什么，有什么证据？ |
| 目标 | 在什么时间，把哪个指标从多少改善到多少？ |
| 高层范围 | 明确包含和不包含什么？ |
| 主要交付物 | 最终要验收哪些能力或文档？ |
| 里程碑 | 哪些日期影响业务、监管或供应商？ |
| 预算与资源边界 | 可以使用多少人、钱、环境和窗口？ |
| 主要风险与假设 | 哪些未知会推翻计划？ |
| 成功标准 | 谁用什么数据确认成功？ |
| 权限和升级路径 | 项目经理可决定什么，何时找发起人？ |

目标示例：

> 在 12 周内为 20 个核心服务建立告警路由、去重和 Runbook，以上线前 4 周为基线，使重复告警占比在上线后连续 4 周下降至少 40%，同时保持漏报事故数不增加。

这个目标同时定义了范围、时间、指标、基线、观察窗口和安全边界。

### 2. 干系人识别与参与

干系人 Stakeholder 是能够影响项目、受项目影响或认为自己受影响的人。

常见做法：

1. 识别发起人、用户、技术团队、运营、安全、采购、财务和供应商。
2. 记录其利益、影响力、成功标准、担忧和沟通偏好。
3. 评估当前参与度与目标参与度。
4. 为关键差距设计行动，例如让值班人员参与验收标准设计。
5. 持续更新，不能只在项目开始时登记一次。

权力高、关注高的干系人通常需要共同决策；权力高但关注低的干系人需要保持满意；不能用矩阵替代真实关系。

### 3. 需求、范围与 WBS

#### 需求

好需求应可追踪到业务目标，并有验收方式。常见类型包括：

- 业务需求：为什么做。
- 干系人需求：不同角色需要什么。
- 解决方案需求：功能和非功能要求。
- 过渡需求：迁移、培训、数据清理和并行运行。

#### 范围基准

范围基准通常由范围说明书、WBS 和 WBS 词典组成。

- 范围说明书定义交付边界和验收。
- WBS 按可交付成果分解工作。
- WBS 词典说明工作包负责人、输入、输出、估算和完成标准。

WBS 不是组织架构，也不是按“前端、后端、测试”简单列部门。更好的一级拆分是：

```text
AIOps 告警治理项目
  -> 1. 项目治理与需求基线
  -> 2. 数据接入与质量
  -> 3. 告警规则与路由
  -> 4. 自动化与安全控制
  -> 5. 可观测性与容量
  -> 6. 测试、演练和上线
  -> 7. 培训、移交和收益验证
```

#### 范围蔓延 Scope Creep

范围蔓延是未经评估和批准的范围增加。它不等于所有变化都错误。正确做法是让变化进入影响分析和决策流程。

### 4. 进度计划与关键路径

#### 活动和依赖

常见依赖关系：

- `FS`，Finish-to-Start，前项完成后后项开始，最常见。
- `SS`，Start-to-Start，前项开始后后项才能开始。
- `FF`，Finish-to-Finish，前项完成后后项才能完成。
- `SF`，Start-to-Finish，前项开始后后项才能完成，较少使用。

`Lead` 是提前量，允许后项提前；`Lag` 是滞后量，要求等待。二者都要有业务依据，不能用来藏缓冲。

#### 三点估算

当不确定性较高时，可以记录：

- `O`，Optimistic，乐观时间。
- `M`，Most Likely，最可能时间。
- `P`，Pessimistic，悲观时间。

PERT 常用期望值：

```text
期望工期 = (O + 4M + P) / 6
```

这不是保证日期，而是把估算区间和不确定性显式化。

#### 关键路径 Critical Path

- **是什么**：决定项目最早完工时间的最长依赖路径。
- **为什么需要**：关键路径上的延迟通常会直接推迟项目结束。
- **怎么工作**：正向计算最早开始和完成，反向计算最晚开始和完成，总浮动为零的活动通常在关键路径上。
- **怎么用或观察**：持续更新实际工期、剩余工期和依赖；关键路径会变化。
- **坏了怎么查**：若里程碑突然延期，检查逻辑依赖、外部依赖、资源约束和实际完成定义是否准确。

`Float` 是浮动时间。自由浮动表示不影响紧后活动的余量，总浮动表示不影响项目完工的余量。

#### 进度压缩

- `Crashing`，赶工：增加资源或成本缩短关键路径活动，只对可通过资源缩短的活动有效。
- `Fast Tracking`，快速跟进：把原本顺序的活动部分并行，会增加返工和协调风险。

项目经理必须先确认活动在关键路径上，再讨论压缩。给非关键活动加人可能对完工日期没有帮助。

### 5. 成本与挣值管理

挣值管理 Earned Value Management，简称 `EVM`，把范围、进度和成本放在同一个测量框架中。

#### 基础量

- `BAC`，Budget at Completion，完工预算。
- `PV`，Planned Value，计划价值：截至当前按计划应完成工作的预算价值。
- `EV`，Earned Value，挣值：截至当前实际完成工作的预算价值。
- `AC`，Actual Cost，实际成本。

#### 常用公式

```text
成本偏差 CV = EV - AC
进度偏差 SV = EV - PV
成本绩效指数 CPI = EV / AC
进度绩效指数 SPI = EV / PV
若当前成本效率持续，完工估算 EAC = BAC / CPI
完工尚需估算 ETC = EAC - AC
完工偏差 VAC = BAC - EAC
```

- `CPI < 1` 表示每花 1 元得到的预算价值不足 1 元。
- `SPI < 1` 表示按预算价值衡量的完成进度落后。
- 项目接近结束时，传统 `SPI` 可能趋近 1，必须同时看里程碑和关键路径。
- 软件项目的 EV 不能用“代码写了 80%”主观申报，应绑定可验收工作包。

### 6. 质量管理

质量不是“测试团队最后找缺陷”，而是交付物满足需求并适合使用。

- 质量规划：定义标准、指标、责任和验证方法。
- 质量保证：改进过程，减少缺陷产生。
- 质量控制：检查具体结果并处理偏差。
- `DoD`，Definition of Done，完成定义：团队对“完成”的共同质量门槛。
- 验收标准：某一需求或交付物被接受的具体条件。

AIOps 项目的完成定义可以包括：

- 代码评审和自动化测试通过。
- 监控、日志和追踪已经接入。
- 权限最小化并通过安全检查。
- 容量与故障演练达到阈值。
- Runbook、回滚和负责人已确认。
- 业务指标可以从可信数据源计算。

### 7. 资源、团队与冲突

资源管理不仅是排人天，还包括技能、设备、环境、供应商和团队健康。

#### 团队容量

可用容量要扣除值班、会议、假期、维护和突发事件。把一个人同时分配到三个“100% 优先级”项目，计划从一开始就是假的。

#### 冲突处理

面对技术冲突，优先：

1. 明确共同目标和决策标准。
2. 分离事实、假设、利益和情绪。
3. 用实验或数据验证关键分歧。
4. 在决策权限内协作解决。
5. 超出权限或影响重大时及时升级。

项目经理不需要替代技术负责人，但必须确保技术决定有负责人、证据、影响分析和期限。

### 8. 沟通管理

沟通计划至少回答：

- 谁需要什么信息？
- 为什么需要？
- 由谁提供？
- 用什么渠道和格式？
- 频率和截止时间是什么？
- 涉及敏感信息时如何授权和留痕？
- 没有回应时怎样升级？

周报应面向决策，而不是罗列忙碌：

```text
总体状态：黄
本周成果：完成 12 个服务告警路由验收
目标偏差：日志接入比基线晚 3 天
主要原因：生产权限审批尚未完成
影响判断：若周三前未批准，将影响灰度开始日期
需要决策：安全负责人确认临时只读权限方案
下一步：完成回滚演练并验证告警漏报指标
```

### 9. 风险、不确定性与 RAID

#### 四类对象

- `Risk` 风险：尚未发生、可能影响目标的不确定事件或条件。
- `Issue` 问题：已经发生并需要处理。
- `Assumption` 假设：计划暂时认为为真的条件。
- `Dependency` 依赖：本项工作依赖的外部交付、决定或资源。

`RAID` 日志把 Risk、Assumption、Issue、Dependency 放在一个可追踪视图中。

#### 风险管理闭环

1. 识别风险和根因，不只写模糊结果。
2. 评估概率、影响、紧迫性和可探测性。
3. 指定风险负责人。
4. 选择响应策略。
5. 定义触发器、应急计划和储备。
6. 持续监控残余风险和次生风险。

威胁的常见响应是规避、减轻、转移、接受或升级；机会可以开拓、提高、分享、接受或升级。

风险示例：

> 因生产日志字段未经数据分级，安全审批可能在第 6 周前无法完成，导致模型训练数据不足并推迟灰度。触发器是第 3 周仍未确认字段清单。负责人是数据负责人；减轻措施是第 1 周完成字段分类并准备脱敏样本；应急方案是先用指标数据上线规则降噪。

### 10. 采购和供应商

技术项目常见合同边界：

- 固定总价：范围清晰时成本可预测，但变更通常需要重新谈判。
- 工料合同：按时间和材料付费，灵活但需要成本上限和透明记录。
- 成本补偿：买方承担更多成本风险，适合高度不确定的研发，但治理要求高。

采购前要明确：

- 自研、购买还是复用现有平台。
- 接口、数据归属、知识产权和退出机制。
- `SLA` 服务等级协议、支持时段和升级路径。
- 验收、付款里程碑和缺陷责任。
- 许可证、续费、数据迁出和厂商锁定风险。

### 11. 整体变更控制

变更控制不是拒绝变化，而是让决策者看到代价和风险。

```text
提出变更
  -> 记录原因和预期价值
  -> 分析范围 / 进度 / 成本 / 质量 / 资源 / 风险影响
  -> 在授权层级批准、拒绝、延期或要求补充证据
  -> 更新基线、需求追踪、风险和沟通
  -> 实施、测试、发布和验证
  -> 保留决定及结果
```

`CCB` 是 Change Control Board，变更控制委员会。小项目不一定需要正式委员会，但必须定义谁有权批准什么级别的变化。

紧急生产变更可以走加速流程，但不能取消最小证据、回滚和事后复盘。

### 12. 收尾、移交与收益确认

项目收尾至少包括：

- 由授权人按标准验收交付物。
- 关闭或移交未完成项、风险和问题。
- 完成合同、资产、账号和财务处理。
- 把监控、告警、Runbook、权限、备份和服务负责人移交运营。
- 记录复盘和可复用知识。
- 在约定观察期确认收益，而不是上线当天宣布成功。

没有运营接收人的系统不算完成交付。

## 敏捷、Scrum、Kanban 与混合交付

### 敏捷不是没有计划

Agile Manifesto 更重视人员互动、可工作的软件、客户协作和响应变化，但没有说流程、文档、合同和计划没有价值。

敏捷改变的是计划方式：

- 长期目标和边界保持清楚。
- 近期工作详细规划，远期工作保留弹性。
- 用短反馈周期验证假设。
- 用可工作的增量衡量进展。
- 把变化成本控制在可承受范围。

### Scrum

Scrum 是用于复杂产品开发的轻量框架。

#### 三种职责

- Product Owner：对产品目标和产品待办项的有效管理负责。
- Scrum Master：对 Scrum 的建立和团队改进负责。
- Developers：对每个 Sprint 产生可用增量负责。

#### 五个事件

- Sprint：固定长度的交付周期。
- Sprint Planning：确定为什么有价值、做什么、怎样做。
- Daily Scrum：开发者检查 Sprint 目标进展并调整计划。
- Sprint Review：与干系人检查增量和环境变化。
- Sprint Retrospective：改进质量和工作方式。

#### 三个工件与承诺

- Product Backlog 对应 Product Goal。
- Sprint Backlog 对应 Sprint Goal。
- Increment 对应 Definition of Done。

常见误区：

- 把 Daily Scrum 变成向经理汇报。
- 把故事点当工时或个人绩效。
- Sprint 内不断插入工作，却不重新协商目标。
- Review 只演示页面，不检查价值和反馈。
- Retrospective 只说感受，没有可验证改进行动。

### Kanban

Kanban 通过定义和可视化工作流、主动管理工作项、持续改进工作流来优化价值流动。

必须关注的流动指标：

- `WIP`，Work in Progress，在制品数量。
- Throughput，单位时间完成的工作项数量。
- Work Item Age，当前未完成工作项已持续多久。
- Cycle Time，工作项从开始到完成的时间。

限制 WIP 的目的不是让人“保持忙碌”，而是减少排队、暴露瓶颈并提高可预测性。

### 怎样选择

| 环境 | 更合适的起点 |
|---|---|
| 范围稳定、外部审批强、变更昂贵 | 预测型 |
| 产品需求不确定、能频繁得到用户反馈 | Scrum 或其他敏捷方法 |
| 运维工单连续到达、优先级经常变化 | Kanban |
| 采购和合规固定、软件功能可迭代 | 混合型 |

方法是工具，不是身份。选择依据应是风险、反馈速度、交付物性质和组织约束。

## IT 项目的内部交付路径

```text
战略目标
  -> 商业论证与项目授权
  -> 产品目标 / 项目目标 / 成功指标
  -> 需求与验收标准
  -> WBS / Backlog / 依赖网络
  -> 设计 / 构建 / 测试
  -> 制品库与发布流水线
  -> 灰度 / 观测 / 决策
  -> 全量 / 回滚 / 运营移交
  -> 用户结果与收益数据
  -> 复盘和下一轮优先级
```

项目状态来自多个真实系统：

- Git 和代码评审记录说明变更发生了什么。
- CI/CD 说明构建、测试、部署和回滚是否成功。
- 监控、日志和链路说明运行结果。
- 工单和事故平台说明用户影响与恢复。
- 财务和采购系统说明成本与合同状态。
- 项目工具说明承诺、负责人、依赖和决策。

只依赖手工周报会产生滞后和美化。AIOps 可以聚合证据、发现异常，但重大项目决策仍需人承担责任。

## 项目管理“配置字典”

项目管理没有必须安装的单一软件。Jira、GitHub Projects、Microsoft Project、Excel、飞书或其他工具都可以承载信息，关键是字段语义和治理一致。

### 工作项字段

| 字段 | 作用 | 常见陷阱 |
|---|---|---|
| ID | 稳定引用工作项 | 用标题代替 ID，改名后无法追踪 |
| Outcome | 说明完成后产生什么结果 | 只写动作，不写结果 |
| Owner | 对推进负责的人 | 填一个群组而没有具体负责人 |
| Acceptance Criteria | 可验证的验收条件 | 写“正常”“优化完成”等模糊词 |
| Dependency | 前置交付或决定 | 只写日期，不写依赖方 |
| Estimate | 当前估算及假设 | 把估算当承诺且不更新 |
| Actual | 实际开始、完成和消耗 | 到周报时才补录 |
| Status | 按统一规则计算的状态 | 负责人凭感觉填红黄绿 |
| Evidence | 测试、监控、评审或验收链接 | 只说完成，没有证据 |

### RAID 字段

| 字段 | 说明 |
|---|---|
| Type | Risk、Assumption、Issue 或 Dependency |
| Description | 具体事件、原因和影响 |
| Probability / Impact | 概率与影响，问题可标记为已发生 |
| Trigger | 何时从观察转为行动 |
| Owner | 负责监控和推动响应的人 |
| Response | 预防、应急、修复或升级动作 |
| Due Date | 下一次动作或决定期限 |
| Residual Risk | 响应后仍保留的风险 |

### 变更请求字段

| 字段 | 说明 |
|---|---|
| Reason | 为什么提出变化 |
| Expected Value | 变化带来什么价值 |
| Options | 不做、部分做、完整做等选项 |
| Impact | 对范围、日期、成本、质量、资源和风险的影响 |
| Recommendation | 基于证据的建议 |
| Decision | 批准、拒绝、延期或补充信息 |
| Approver | 在授权范围内作决定的人 |
| Baseline Updates | 决定后需要更新的基线和文档 |
| Verification | 实施后怎样证明结果正确 |
| Rollback | 失败或超阈值时怎样恢复 |

## DevOps、SRE 与项目治理

### 上线不是项目终点

软件交付必须同时满足速度和稳定性。项目计划至少包含：

- 环境、制品和配置可重复。
- 自动化测试与安全检查。
- 灰度策略和停止条件。
- 监控、日志、追踪和业务指标。
- 回滚或前滚修复能力。
- SLO、错误预算和事故响应。
- 运维值班、Runbook 和知识移交。

### DORA 指标

当前 DORA 使用五项软件交付性能指标：

- Change Lead Time，变更前置时间。
- Deployment Frequency，部署频率。
- Failed Deployment Recovery Time，失败部署恢复时间。
- Change Fail Rate，变更失败率。
- Deployment Rework Rate，部署返工率。

这些指标用于团队和系统持续改进，不应被用作个人排名。跨产品比较时必须说明上下文差异。

### SLO 与错误预算

`SLO` 是 Service Level Objective，服务水平目标。错误预算表示目标允许的失败空间。

当错误预算消耗过快时，可以触发：

- 暂缓高风险功能。
- 优先修复可靠性缺陷。
- 增加容量或故障演练。
- 调整发布策略。

项目经理应把可靠性工作放进正式范围和优先级，而不是把它留给运维团队“上线后再补”。

## AIOps 项目管理场景

### 指标、日志和链路

- 明确数据所有者、保留期、质量和访问权限。
- 为缺失、延迟、重复和字段变化建立质量指标。
- 记录采集成本和容量假设。

### 告警治理

- 基线包括告警总量、重复率、有效率、确认时间和升级次数。
- 目标不能只写“降噪”，还要防止漏报和延迟发现。
- 每条核心告警要有负责人、Runbook 和升级路径。

### 自动化处置

- 把只读诊断、低风险修复和高风险生产变更分级。
- 定义审批、幂等、超时、熔断、审计和回滚。
- 先在测试环境和影子模式验证，再逐步放大影响范围。

### 异常检测和模型

- 区分离线指标、在线业务结果和人工接受度。
- 跟踪数据漂移、模型版本、阈值、误报和漏报。
- 模型发布必须可回滚，人工决策必须保留证据。

### 知识库和运维助手

- 定义来源、权限、时效、引用和敏感数据边界。
- 把回答正确率、检索命中、拒答和升级人工纳入验收。
- 不能用“看起来像对”代替可追溯证据。

## 生产设计检查

### 状态与一致性

- 需求、代码、配置、制品和部署版本如何关联？
- 项目工具与 Git、CI/CD、监控数据发生冲突时，以谁为事实来源？
- 基线更新是否有授权和审计？
- 指标口径变化后，历史趋势怎样保持可比？

### 高可用与业务连续性

- 项目关键人员、供应商、环境和审批是否存在单点？
- 关键资料和凭据是否可恢复？
- 上线失败后业务如何继续运行？
- 灾备和回滚是否真正演练，而不是只写文档？

### 容量与性能

- 团队容量是否扣除了值班和维护？
- 测试环境、流水线、数据平台和监控系统能否承受峰值？
- 估算依据、增长率和安全余量是什么？
- 性能指标是平均值还是 P95/P99 等尾延迟？

### 安全与合规

- 数据怎样分级、脱敏、授权和审计？
- 生产权限是否最小化并定期复核？
- 第三方组件、许可证和供应链风险怎样管理？
- 紧急变更如何留痕并在事后复核？

### 升级与回滚

- 升级前后版本兼容矩阵是什么？
- 数据库或不可逆数据变更如何处理？
- 停止条件、回滚点和最长决策时间是什么？
- 回滚后怎样验证业务与数据一致性？

### 可观测性

同时观察三层：

1. 交付流：排队时间、周期时间、WIP、阻塞年龄、部署表现。
2. 系统运行：可用性、延迟、错误、容量、告警和事故。
3. 业务价值：用户采用、效率、成本、风险和满意度。

项目仪表盘应帮助发现系统问题，不能诱导团队为了“变绿”修改数据口径。

## 入门实验：建立可计算的 AIOps 项目计划

目标：不用购买项目管理软件，在本地建立项目章程、活动网络、关键路径和挣值快照。

### 前置条件

- Windows PowerShell 5.1 或 PowerShell 7。

### 第 4 步：编写关键路径脚本

新建 `critical-path.ps1`：

```powershell
$tasks = Import-Csv .\activities.csv
$state = @{}

# 正向计算每个活动最早可以开始和完成的时间。
foreach ($task in $tasks) {
    $predecessorIds = @(
        $task.predecessors -split ';' |
            Where-Object { -not [string]::IsNullOrWhiteSpace($_) }
    )

    foreach ($id in $predecessorIds) {
        if (-not $state.ContainsKey($id)) {
            throw "活动 $($task.id) 的前置活动 $id 不存在或顺序错误"
        }
    }

    $earliestStart = 0
    if ($predecessorIds.Count -gt 0) {
        $earliestStart = (
            $predecessorIds |
                ForEach-Object { $state[$_].EF } |
                Measure-Object -Maximum
        ).Maximum
    }

    $duration = [int]$task.duration_days
    $state[$task.id] = [ordered]@{
        Id = $task.id
        Name = $task.name
        Duration = $duration
        Pred = $predecessorIds
        ES = [int]$earliestStart
        EF = [int]$earliestStart + $duration
        LS = 0
        LF = 0
        Float = 0
    }
}

$projectDuration = (
    $state.Values.EF |
        Measure-Object -Maximum
).Maximum

# 建立后继关系，供反向计算最晚开始和完成时间。
$successors = @{}
foreach ($task in $tasks) {
    $successors[$task.id] = @()
}
foreach ($task in $tasks) {
    foreach ($id in $state[$task.id].Pred) {
        $successors[$id] += $task.id
    }
}

# 从最后一个活动向前计算，不延误项目时允许的最晚时间。
for ($index = $tasks.Count - 1; $index -ge 0; $index--) {
    $id = $tasks[$index].id
    $latestFinish = $projectDuration

    if ($successors[$id].Count -gt 0) {
        $latestFinish = (
            $successors[$id] |
                ForEach-Object { $state[$_].LS } |
                Measure-Object -Minimum
        ).Minimum
    }

    $state[$id].LF = [int]$latestFinish
    $state[$id].LS = [int]$latestFinish - $state[$id].Duration
    $state[$id].Float = $state[$id].LS - $state[$id].ES
}

$result = foreach ($task in $tasks) {
    $item = $state[$task.id]
    [pscustomobject]@{
        Id = $item.Id
        Name = $item.Name
        ES = $item.ES
        EF = $item.EF
        LS = $item.LS
        LF = $item.LF
        Float = $item.Float
        Critical = $item.Float -eq 0
    }
}

$result | Format-Table -AutoSize
"项目最短工期: $projectDuration 天"
"关键路径: " + (($result | Where-Object Critical).Id -join ' -> ')
```

运行：

```powershell
powershell -ExecutionPolicy Bypass -File .\critical-path.ps1
```

预期结果：

```text
项目最短工期: 14 天
关键路径: A -> C -> D -> E -> F
```

活动 `B` 的 `Float` 应为 `1`，其他关键活动的 `Float` 应为 `0`。

### 第 5 步：计算挣值

假设完工预算 `BAC` 为 100000 元，当前按计划应完成 60000 元工作，实际验收完成 50000 元工作，实际花费 65000 元：

```powershell
$bac = 100000
$pv = 60000
$ev = 50000
$ac = 65000

$cv = $ev - $ac
$sv = $ev - $pv
$cpi = $ev / $ac
$spi = $ev / $pv
$eac = $bac / $cpi

[pscustomobject]@{
    CostVariance = $cv
    ScheduleVariance = $sv
    CPI = [math]::Round($cpi, 2)
    SPI = [math]::Round($spi, 2)
    EAC = [math]::Round($eac, 2)
} | Format-List
```

预期：

- `CostVariance` 为 `-15000`，表示超出已完成工作对应的预算。
- `ScheduleVariance` 为 `-10000`，表示按预算价值衡量的进度落后。
- `CPI` 约为 `0.77`。
- `SPI` 约为 `0.83`。
- 若当前成本效率持续，`EAC` 约为 `130000`。

结论不能只写“红色”。要说明偏差、原因假设、趋势、影响和需要的决定。

## 常见失败与排障

### “完成 90%”持续三周

可能原因：完成定义不清、剩余工作隐藏、依赖未解决。

检查：

1. 按可验收交付物重估剩余工作。
2. 查看阻塞年龄和未关闭缺陷。
3. 验证环境、权限和外部接口。
4. 不使用已经花费的工时推导完成度。

### 项目一直绿，最后突然红

可能原因：状态由负责人主观填写、风险没有触发器、数据更新滞后。

修复：

- 定义红黄绿客观规则。
- 接入里程碑、关键路径、风险和交付流水线证据。
- 黄色表示需要管理行动，不是失败或追责。

### 敏捷团队每个 Sprint 都完不成

检查：

- Sprint 目标是否频繁被插单破坏。
- WIP 是否过高。
- 待办项是否太大或验收不清。
- 依赖是否在 Sprint 外等待。
- 速度是否被当成必须增长的绩效指标。

### 供应商说“已经交付”，业务不接受

检查合同和需求追踪中的：

- 交付物定义。
- 验收标准和验收人。
- 测试环境与数据。
- 缺陷等级和修复期限。
- 付款与验收里程碑。

### 上线成功但业务没有收益

检查：

- 成功标准是否只衡量输出。
- 用户是否采用新流程。
- 指标基线和数据口径是否可信。
- 运营团队是否持续使用和维护。
- 外部变化是否使原商业论证失效。

### 团队天天加班仍然延期

检查系统问题：

- 优先级是否超过容量。
- 多项目切换是否造成大量等待。
- 关键技能是否单点。
- 需求和验收是否反复变化。
- 测试、环境和审批是否成为瓶颈。

持续加班不是可持续的进度压缩策略。

## 生产事故场景

### 场景

距离项目验收两天，告警治理平台灰度后出现路由错误：部分数据库告警被发给错误团队。业务要求继续上线以保证里程碑，运维要求立即回滚。

### 处理顺序

1. **控制影响**：暂停扩大灰度，关闭错误自动路由，恢复原通知链路。
2. **保存证据**：保留规则版本、部署记录、受影响告警、时间线和人工操作。
3. **确认影响范围**：哪些服务、告警等级、值班组和时间窗受影响，是否存在漏报。
4. **形成假设**：标签映射错误、CMDB 数据过期、规则优先级覆盖或缓存未刷新。
5. **逐项验证**：用同一条测试告警对比输入标签、规则命中、路由结果和通知记录。
6. **选择修复**：优先恢复安全状态，再修复映射并补回归测试。
7. **评估项目影响**：验收条件未满足，应正式记录问题和里程碑影响。
8. **决定与沟通**：由有权限的负责人决定延期、缩小范围或重新灰度。
9. **复盘**：补充配置验证、CMDB 新鲜度监控、灰度门禁和回滚演练。

面试时不要说“先重启试试”。先控制影响、建立事实，再判断根因和修复。

## 生产设计题：90 天交付 AIOps 告警治理平台

### 题目

为 100 个服务设计 90 天项目计划。要求接入指标与日志、减少重复告警、建立值班路由和 Runbook，并支持低风险自动化。

### 回答框架

#### 1. 先澄清价值和约束

- 当前告警量、重复率、误报、漏报和 MTTA 基线。
- 服务分级、生产权限、安全和合规边界。
- 团队、预算、现有平台和固定里程碑。
- 成功由谁在什么观察期确认。

#### 2. 分阶段交付

- 第 1-2 周：章程、基线、服务分级、数据权限和试点选择。
- 第 3-4 周：数据质量、统一标签、路由模型和验收标准。
- 第 5-7 周：20 个试点服务接入、去重、Runbook 和仪表盘。
- 第 8-9 周：故障注入、容量、安全、回滚和运营演练。
- 第 10-11 周：分批扩大到 100 个服务。
- 第 12-13 周：收益观察、遗留项移交和复盘。

#### 3. 关键治理

- 发起人确认目标和跨团队资源。
- 服务 Owner 对告警和 Runbook 负责。
- 生产自动化按风险分级审批。
- 变更必须关联规则、代码、制品、部署和验证证据。

#### 4. 指标

- 业务：重复告警占比、有效告警率、MTTA、漏报事故。
- 交付：周期时间、WIP、阻塞年龄、部署性能。
- 系统：延迟、错误、容量、规则命中和通知成功率。
- 安全：越权、审计完整性和自动化回滚成功率。

#### 5. 风险和退出条件

- 数据质量不足时先完成标签治理，不扩大模型范围。
- 漏报超过阈值时停止扩大灰度并恢复原规则。
- 自动化动作不能证明幂等和回滚时，只保留建议模式。

优秀答案要把项目管理、架构、交付流水线和生产可靠性连起来。

## 面试问题与回答指导

### 1. 项目和运营有什么区别？

回答主线：项目有明确起止时间并产生独特成果；运营持续提供重复服务。说明平台建设结束后会转入运营，并需要移交。

追问：谁确认项目结束后的收益？运营发现设计缺陷由谁处理？

### 2. 怎样制定一个可信计划？

回答主线：先明确范围和验收，建立 WBS，识别活动、依赖、资源和风险，估算后计算关键路径，再用历史数据和团队评审校准。

追问：估算没有历史数据怎么办？关键路径变化怎样发现？

### 3. 什么是关键路径？

回答主线：决定最早完工时间的最长依赖路径，总浮动通常为零。关键路径上的延迟通常影响项目结束。

追问：增加资源一定能缩短吗？资源约束如何影响网络计划？

### 4. 风险和问题有什么区别？

回答主线：风险尚未发生，问题已经发生。风险需要概率、影响、负责人、触发器和响应；问题需要修复、升级和期限。

追问：一个风险发生后怎样更新 RAID？残余风险是什么？

### 5. 如何处理范围变更？

回答主线：记录价值和原因，分析各基线与风险影响，由授权人决定，批准后更新计划并验证结果。变更控制是明智选择，不是拒绝变化。

追问：紧急生产变更怎么办？小变更是否都需要 CCB？

### 6. 敏捷和瀑布怎样选择？

回答主线：根据需求不确定性、反馈速度、变更成本、合规和交付物性质裁剪。多数企业 IT 项目适合混合型。

追问：固定日期的敏捷项目怎样管范围？硬件采购如何与软件 Sprint 对齐？

### 7. 怎样判断项目进度？

回答主线：看可验收成果、关键路径、里程碑、流动指标和实际证据，不只看主观百分比。必要时使用 EVM。

追问：SPI 正常为什么仍可能延期？故事点能否跨团队比较？

### 8. 技术负责人不同意你的排期怎么办？

回答主线：先理解估算假设和技术风险，用目标、依赖、实验和选项对齐；项目经理不能凭职位覆盖技术事实，重大取舍由对应权限人决定。

追问：如果发起人坚持不现实日期怎么办？

### 9. 如何管理 AIOps 模型项目？

回答主线：把数据质量、离线评估、在线业务结果、模型版本、漂移、人工反馈、安全和回滚都纳入范围。

追问：准确率高为什么线上无价值？误报和漏报怎样权衡？

### 10. 项目上线失败如何处理？

回答主线：先停止扩散和恢复安全状态，再保存证据、确认影响、验证假设、修复和复盘；同时按治理流程记录里程碑和风险影响。

追问：什么时候回滚，什么时候前滚修复？谁有决定权？

## 面试表达模板

### 30 秒版本

> 我把项目管理理解为价值、交付和治理的整合。先用章程明确目标、范围和成功指标，再通过 WBS、依赖和关键路径建立计划，用 RAID、变更控制和客观交付数据处理不确定性。软件项目还必须把 CI/CD、可观测性、SLO、灰度和回滚纳入完成定义，最终由业务结果而不是任务数量判断是否成功。

### 3 分钟项目故事

按以下顺序组织：

1. 背景和可量化问题。
2. 目标、范围、约束和你的真实职责。
3. 关键依赖和最高风险。
4. 你如何组织计划、沟通和技术验证。
5. 发生了什么偏差或冲突。
6. 你怎样用证据形成选项并推动决定。
7. 技术结果和业务结果。
8. 复盘后改变了什么。

不要虚构预算、团队规模或生产指标。不确定的数据要明确说“当时未保留精确数字”。
