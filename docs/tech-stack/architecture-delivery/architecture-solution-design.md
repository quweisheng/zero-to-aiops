# 系统架构与技术方案设计深讲

> 学习目标：能把模糊业务需求整理成约束、质量目标、候选方案和可实施设计，完成一份带容量、高可用、安全、可观测性、上线与回滚的可行性报告，并在评审中解释取舍。

## 官方资料

- [AWS Well-Architected Framework](https://docs.aws.amazon.com/wellarchitected/latest/framework/welcome.html)
- [Google Cloud Architecture Framework](https://cloud.google.com/architecture/framework)
- [Kubernetes Production Environment](https://kubernetes.io/docs/setup/production-environment/)
- [C4 Model](https://c4model.com/)
- [Architecture Decision Records](https://adr.github.io/)

这些资料提供不同视角，不是要求照搬某家云。本文把可靠性、安全、成本、运维和架构表达整理为适合 AIOps 平台岗位的通用方法。

## 官方知识地图

```text
需求与约束
  -> 功能需求 / 非功能需求
  -> 当前状态与数据基线
  -> 候选方案与可行性验证
  -> 总体架构 / 详细设计
  -> 容量 / 高可用 / 安全 / 可观测性
  -> 实施计划 / 验收 / 回滚 / 运维移交
  -> ADR 与持续复盘
```

`ADR` 是 Architecture Decision Record，即架构决策记录；它说明当时为什么选择某方案，而不只是记录最终结果。`HLD` 是 High-Level Design，总体设计；`LLD` 是 Low-Level Design，详细设计。

## 场景开场

业务说：“我们要一套稳定的 Kubernetes 平台，能支持两百个服务，最好高可用。”这句话还不能直接变成采购清单。你必须追问并发、峰值、恢复目标、合规边界、现有机房、团队能力和预算，才能判断方案是否可行。

## 一句话人话版

架构设计就是把业务目标、现实约束和故障风险变成一套可解释、可验证、可上线、可回滚的技术取舍。

## 小白可能会问

- 架构图画得完整，为什么还不能算技术方案？
- “高可用”到底要问哪些数字？
- 可行性报告与详细设计有什么区别？
- 没有五年经验，怎么证明自己会做方案？

## 为什么要学

这份岗位把“整体架构设计、可行性报告、需求沟通和方案制定”放在任职要求之前，说明技术栈只是材料，方案能力才是把材料组合成业务结果的方法。AIOps 平台还必须说明数据从哪里来、怎样告警、怎样自动化以及自动化失败如何止损。

## 系统架构与技术方案设计是什么

系统架构定义组件边界、责任、接口、数据流和部署关系。技术方案进一步回答版本、容量、权限、监控、故障、迁移、成本与验收。可行性报告则在投入实施前证明：方案在技术、资源、进度、风险和组织能力上可以落地。

## 它解决什么问题

- 防止团队在需求尚未澄清时直接选产品。
- 用数字定义“快、稳定、可恢复”。
- 比较多个候选方案并保留决策依据。
- 让开发、运维、安全和业务对同一边界达成共识。
- 提前设计监控、故障域、回滚和运维交接。

## 核心原理

### 功能需求、非功能需求与约束

- **是什么**：功能需求说明系统做什么；非功能需求说明性能、可靠性、安全等质量；约束说明不能改变的边界。
- **为什么需要**：同一个功能在不同可用性和合规要求下，成本可能相差数倍。
- **怎么工作**：把口头描述改写成可测量条件，例如“月可用性 99.9%、P95 小于 300 ms”。
- **怎么看或怎么用**：建立需求追踪表，每项需求必须有负责人、验收方法和优先级。
- **坏了怎么查**：评审争论技术细节时，回到原始业务目标和约束，检查是否存在未确认假设。

### SLI、SLO、RTO 与 RPO

- **是什么**：SLI 是服务水平指标；SLO 是目标；RTO 是故障后允许多久恢复；RPO 是最多允许丢失多久的数据。
- **为什么需要**：没有数字的“高可用”和“不能丢数据”无法设计也无法验收。
- **怎么工作**：先选用户可感知指标，再定义统计窗口、目标和错误预算，最后反推冗余与灾备。
- **怎么看或怎么用**：方案中写明数据源、计算公式、告警阈值和演练频率。
- **坏了怎么查**：若指标长期达标但用户仍投诉，说明 SLI 可能没有代表真实体验。

### 容量模型与故障域

- **是什么**：容量模型估算峰值资源；故障域表示一次故障会同时影响的范围。
- **为什么需要**：平均负载不能代表峰值，三副本若都在同一节点也不是真正高可用。
- **怎么工作**：使用峰值 QPS、单实例吞吐、增长率和冗余系数计算容量，再把副本分散到节点、机架或可用区。
- **怎么看或怎么用**：用压测验证单实例基线，用 topology spread、反亲和和存储副本验证分散。
- **坏了怎么查**：容量不足先确认假设与真实数据是否偏离；故障扩大则检查副本是否共享网络、电源、存储或控制面。

### 候选方案、PoC 与 ADR

- **是什么**：候选方案用于比较，PoC 是 Proof of Concept 概念验证，ADR 保存决策与取舍。
- **为什么需要**：避免凭产品熟悉度拍板，也避免 PoC 只展示“能安装”。
- **怎么工作**：用同一组硬性条件和评分维度比较方案，对最高风险假设设计 PoC。
- **怎么看或怎么用**：ADR 记录背景、选项、决定、后果和重新评估条件。
- **坏了怎么查**：实施后条件变化时回看 ADR，不把旧决定当永久真理。

### 上线、回滚与运维移交

- **是什么**：上线计划描述变更步骤，回滚定义停止条件与恢复动作，移交保证系统可持续运维。
- **为什么需要**：安装成功不是交付完成，未知告警和无法恢复的备份都会变成生产风险。
- **怎么工作**：先建立基线，再灰度实施，按技术与业务指标验证，越过阈值立即停止或回滚。
- **怎么看或怎么用**：必须有 Runbook、仪表盘、告警、权限、备份恢复演练和负责人。
- **坏了怎么查**：上线异常先冻结继续扩散，保存证据，按预先定义条件回滚，而不是现场争论。

## 架构和数据流

```text
业务目标
  -> 可测量需求与约束
  -> 当前架构和数据基线
  -> 候选方案
  -> 风险最高假设的 PoC
  -> HLD 与 LLD
  -> 分阶段实施
  -> 验收、回滚与运维移交
  -> 运行数据推动下一次 ADR
```

```powershell

```

## 命令 / 配置 / API 字典

| 名称 | 作用 | 常用写法 | 正常结果 | 常见坑 |
|---|---|---|---|---|
| SLI | 测量服务表现 | 成功请求 / 有效请求 | 能从稳定数据源计算 | 只监控机器，不代表用户体验 |
| SLO | 定义目标 | 30 天可用性 99.9% | 有窗口、范围和责任人 | 把 100% 当默认目标 |
| RTO | 定义恢复时限 | 核心服务 30 分钟 | 灾备演练可验证 | 只写数字，没有恢复步骤 |
| RPO | 定义数据丢失上限 | 配置数据 5 分钟 | 备份频率和复制满足目标 | 把备份成功等同可恢复 |
| ADR | 保存架构决定 | Context/Options/Decision/Consequences | 后人能理解当时取舍 | 只写“选择了什么” |

## 在 AIOps 中的作用

AIOps 架构必须同时设计数据源、标签、时间、拓扑、模型/规则、告警、自动化和审计。自动修复要定义触发条件、权限、幂等、超时、人工审批、熔断与回滚；否则“智能化”会把误判放大成变更事故。

## 入门实验：为多集群平台写一页方案并自动检查

### 实验目标

完成一页最小可行性报告，确保关键章节不是空标题，并能说明一个核心取舍。

### 实验步骤

1. 使用上面的模板填写 `solution-lab/feasibility.md`。
2. 场景限定为：两套 Kubernetes 集群、50 个服务、需统一镜像和监控，核心服务 RTO 30 分钟、RPO 5 分钟。
3. 至少比较“单管理平台多集群”和“每集群独立管理”两种方案。
4. 执行检查：

```powershell
$required = '业务目标与范围','当前状态与约束','非功能目标','候选方案与取舍','目标架构与容量','安全、可观测性与运维','实施、验收与回滚','风险、成本与结论'
$text = Get-Content .\solution-lab\feasibility.md -Raw -Encoding utf8
$missing = $required | Where-Object { $text -notmatch [regex]::Escape($_) }
if ($missing) { throw "缺少章节: $($missing -join ', ')" } else { 'PASS: 必需章节齐全' }
```

### 验证结果

终端输出 `PASS: 必需章节齐全`；文档中每个目标都有数字和验收方法，每个主要风险都有负责人或缓解动作。

### 如果没有成功

1. 检查文件路径和 UTF-8 编码。
2. 若章节存在但名称不同，统一标题或调整检查列表。
3. 章节齐全但内容空泛时，补数据来源、负责人、验收命令和失败处理。
4. 无法比较方案时，说明需求和约束还没有问清，不要先画最终架构图。

## 常见故障排查

| 现象 | 先检查 | 处理思路 |
|---|---|---|
| 方案评审反复争论产品 | 目标、约束、决策标准 | 先确认问题，再比较方案 |
| 容量上线即不足 | 峰值、增长率、压测基线、冗余 | 修正模型并建立扩容阈值 |
| 三副本仍整体中断 | 故障域、共享依赖 | 分散副本并消除单点 |
| 监控很多但无法验收 | SLI 与业务目标关系 | 用用户路径定义 SLO |
| 回滚方案无法执行 | 数据兼容、备份、权限、演练 | 上线前实际演练并记录时长 |

## 面试怎么讲

我会先把业务目标翻译成范围、约束和可测量的非功能指标，再用容量基线和故障域设计候选架构。对风险最高的假设做 PoC，用 ADR 记录取舍，方案中同时包含安全、可观测性、成本、分批实施、验收和回滚，最后通过运行数据复盘。
